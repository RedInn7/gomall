## 阅读时的几个原则

1. **先看代码再看博客**：每篇文末都列了相关代码位置，先 `git checkout` 看实现，再读博客的"为什么这样做"。
2. **跟着画时序图**：第 3 篇和第 4 篇的关键场景都是并发竞态，文字+代码看不懂时自己画一遍 sequence diagram，思路就通了。
3. **跑一遍单元测试**：每个特性都有 `*_test.go`，亲手跑 + 改阈值，比看文档收益高 10 倍。
4. **复现一次故障**：第 4 篇有一个 RMQ 关停演练，强烈推荐做一遍——理论上"Outbox 解决最终一致性"是 1 行话，亲手让它工作起来才是真理解。

## 配套压测脚本

`stressTest/` 下有完整的 k6 压测脚本，每个特性对应一个：

```bash
cd stressTest
./setup_token.sh        # 注册一个测试用户 + 拿 token
./run_all.sh            # 一键跑所有压测，结果在 results/ 目录
```

完整压测报告见 `stressTest/REPORT.md`，跑出来的关键数字：
- 优化版订单列表 vs 旧版深分页：**吞吐差 7000 倍、p95 差 3200 倍**
- 幂等中间件：**755K 重复请求 → 1 笔订单**
- 滑动窗口：**配置 3/秒/用户，15s 实际通过 46 次（期望 45）**
- 库存零超发：**500 并发抢 100 件，实测恰好 100**
