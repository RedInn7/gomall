\section{业务出发点：一致性意味着什么}
% ============================================================

\begin{frame}{一致性是\textbf{业务事实}，不是技术口号}
\begin{itemize}
  \item 业务上的"一致"：库存数、订单状态、搜索结果、用户能看到的页面，四者要对得上。
  \item gomall 里至少有四个独立存储：MySQL、Redis、RabbitMQ、Elasticsearch。
  \item 任何一对存储之间都可能"一边成功一边失败"。
  \item 业务能容忍的不是"零延迟一致"，而是"最终一致 + 看得见的窗口"。
\end{itemize}
\srcnote{cmd/main.go:29--47 初始化序列里 4 个存储依次进场}
\end{frame}

\begin{frame}{gomall 里业务侧最在意的三组一致性}
\begin{tabular}{@{}lll@{}}
\toprule
一致性对 & 业务后果 & gomall 的兜底 \\
\midrule
DB $\leftrightarrow$ MQ      & 下游收不到事件 / 搜索索引腐烂 & \textbf{Outbox} (deck 主线) \\
DB $\leftrightarrow$ Cache   & 用户看到旧价 / 旧库存 & 延迟双删 + SETNX 回源锁 \\
跨服务步骤                    & 半成品订单、超卖、漏退款 & 协同式 Saga + 补偿事件 \\
\bottomrule
\end{tabular}
\vspace{2mm}
\small 三组一致性问题各有典型出错形态。本 deck 只挑这三组，因为 gomall 实际改过、修过、复盘过。
\srcnote{service/order.go、service/order\_cancel.go、service/product.go 三处对应改动}
\end{frame}

  \draw[arrow-bad]  (52mm,-20mm)-- node[above,font=\tiny\color{cBad}]{timeout}  (0,-20mm);
  \node[font=\tiny,text=cBad] at (39mm,-26mm) {DB 有单 / MQ 没事件 $\Rightarrow$ 搜索索引、库存对账、风控全断流};
\end{tikzpicture}
\vspace{2mm}
\textbf{反过来}：先 publish 再 commit，commit 失败时下游会收到"幽灵订单"，更糟。
\srcnote{docs/blog/04-outbox-saga.md \S 这就是双写问题}
\end{frame}

\begin{frame}{为什么 gomall 不能"两个 try / catch 套一下"}
\begin{itemize}
  \item DB 事务回滚不会回滚已经发出去的 MQ 消息。
  \item MQ 投递失败可以重试，但应用进程崩了之后重试上下文也没了。
  \item 2PC / XA 在 MQ 那一侧支持很差，RabbitMQ 官方没有 XA 资源管理器。
  \item 业务无法接受"先停止下单、等运维手动对账"。
\end{itemize}
\vspace{1mm}
\keypoint{业务层结论：必须把"事件"也当成 DB 行写进去，让事务统一管。}
\srcnote{deck 04 \S 双写问题 / 业界标准答案：Transactional Outbox}
\end{frame}

  \draw[arrow] (txm) -- (out);

\vspace{2mm}
\small \textbf{横轴}：左强一致 $\to$ 右最终一致；左侧阻塞重、右侧补偿轻。\\
gomall 选 \textbf{Outbox + 协同 Saga}（右下角组合）：单 DB 事务 + 异步事件 + 业务码幂等。

\end{frame}

\begin{frame}{六种路线参数对比}
\begin{tabular}{@{}lcccc@{}}
\toprule
路线 & 一致性 & 锁/阻塞 & 运维成本 & gomall 适配 \\
\midrule
2PC / XA          & 强 (ACID)  & 协调者持锁全程 & 高 (需 XA broker) & 不适配 RMQ \\
3PC               & 强         & 引入预提交相位 & 高 (3 阶段超时窗) & 业界几乎不用 \\
TCC               & 强 (业务级) & Try 锁定资源   & 中 (每接口 3 个)  & 资金/库存可，业务接口翻 3 倍 \\
Saga (编排)       & 最终一致   & 中心 Coordinator& 中 (多一个组件)  & 5 步以上链路 \\
Saga (协同)       & 最终一致   & 无中心          & 低 (事件即 API)   & \textbf{gomall 选用} \\
Outbox            & 最终一致   & 单 DB 事务      & 低 (一张表 + 轮询)& \textbf{gomall 选用} \\
事务消息 (RocketMQ)& 最终一致   & Half-Msg 二阶段 & 中 (RocketMQ 专属)& 用 RocketMQ 时首选 \\
\bottomrule
\end{tabular}
\vspace{2mm}
\small 一个判断口诀：\textbf{资金账户}走 TCC（强一致 + 业务可接受 Try 锁定）；\textbf{订单/库存/搜索}走 Outbox + Saga（最终一致 + 业务可接受秒级窗口）。
\srcnote{对照 service/inventory/ (TCC 风格预扣) 与 service/outbox/ (最终一致)}
\end{frame}

\begin{frame}{为什么 2PC / XA 在互联网业务里几乎绝迹}
\begin{itemize}
  \item \textbf{协调者单点}：协调者宕机时所有参与者都卡在 prepared 阶段持锁。
  \item \textbf{同步阻塞}：prepare 到 commit 之间所有参与者都不能让出资源，p99 被拖死。
  \item \textbf{异构资源支持差}：MySQL 有 XA、Redis 没有、RabbitMQ 没有、ES 没有 —— 四个存储有三个不支持。
  \item \textbf{网络分区时数据被冻结}：见 Jim Gray 论文与 CAP 推论。
\end{itemize}
\vspace{1mm}
\keypoint{2PC 强一致的代价是\textbf{可用性折半}。gomall 选 AP 路线，把不变量写在业务码里。}
\srcnote{docs/blog/04-outbox-saga.md \S 为什么不用 2PC}
\end{frame}

\begin{frame}{TCC：业务接口翻 3 倍的代价}

\section{Transactional Outbox：业务变更 + 事件落表}
% ============================================================

\begin{frame}{Outbox 的核心一句话}
\begin{tikzpicture}[node distance=6mm and 10mm]
  \node[node-box] (biz) {业务行\\order/product};
  \node[node-box,right=of biz,minimum width=34mm,node-warn]
       (tx)  {同一个 TX\\\texttt{INSERT order}\\\texttt{INSERT outbox\_event}};
  \node[node-box,below=of tx,node-ok] (pub) {Publisher\\轮询 pending};
  \node[node-box,right=of pub] (mq) {RabbitMQ};
  \node[node-box,right=of mq] (cons) {Consumer\\(ES / 库存 / 风控)};
  \draw[arrow] (biz) -- (tx);
  \draw[arrow] (tx)  -- node[right,font=\tiny]{FetchBatch} (pub);
  \draw[arrow] (pub) -- node[above,font=\tiny]{publish + MarkSent} (mq);
  \draw[arrow-async] (mq) -- (cons);
\end{tikzpicture}
\vspace{2mm}
\small \textbf{业务变更 + 事件}写在同一个 DB 事务里 —— 要么都成功，要么都失败。 \\
"投递给 MQ" 这件事被下沉到一个独立的轮询循环，业务请求不再等 MQ ack。
\srcnote{service/order.go::CreateOrder / service/order\_cancel.go::CancelUnpaidOrder}
\end{frame}

\begin{frame}[fragile]{outbox\_event 表结构：业务字段就是状态机字段}
\begin{lstlisting}[language=Go,firstnumber=17,
  caption={\srcnote{repository/db/model/outbox.go:17--37}}]
type OutboxEvent struct {
    gorm.Model
    AggregateType string    // "order" / "product"  业务实体类型
    AggregateID   uint      // 业务实体 id —— 下游幂等 key 之一
    EventType     string    // "OrderCreated" / "OrderCancelled"
    RoutingKey    string    // "order.created"      投递到哪条 routing
    Payload       string    // JSON body —— 包含一切下游需要的字段
    Status        int       // 1 pending / 2 sent / 3 dead
    Attempts      int       // 已尝试投递次数
    NextRetryAt   time.Time // 指数退避后的下一次允许时间
    LastError     string    // 最后一次失败原因，截断 500 字
}
\end{lstlisting}
\vspace{1mm}
\small 几个关键设计：\textbf{Status} 是业务码（pending/sent/dead），不是布尔；\textbf{NextRetryAt} 让退避不依赖外部调度；\textbf{LastError} 留给值班同学复盘。
\end{frame}

\begin{frame}[fragile]{CancelUnpaidOrder：业务变更 + 事件 同事务}
\begin{lstlisting}[language=Go,firstnumber=30,
  caption={\srcnote{service/order\_cancel.go:30--51}}]
err = baseDao.DB.Transaction(func(tx *gorm.DB) error {
    ok, err := dao.NewOrderDaoByDB(tx).CloseOrderWithCheck(orderNum)
    if err != nil {
        return err
    }
    if !ok {
        return nil    // 已经被关过，幂等 no-op
    }
    closed = true
    return dao.NewOutboxDaoByDB(tx).Insert(
        "order", "OrderCancelled", "order.cancelled", order.ID,
        events.OrderCancelled{
            OrderID: order.ID, OrderNum: orderNum,
            UserID:  order.UserID, ProductID: order.ProductID,
            Num:     order.Num, Reason: "timeout",
        },
    )
})
\end{lstlisting}
\vspace{1mm}
\small 逐行讲（核心 4 点）：\texttt{Transaction(...)} 闭包返 err 自动回滚；\texttt{CloseOrderWithCheck} 只对 UnPaid 生效，二次调用 no-op 就是 Saga 步骤的业务幂等键；\texttt{NewOutboxDaoByDB(tx)} 让事件 INSERT 跟订单 UPDATE 共享行锁版本；闭包出去之后关单与事件一起可见或一起未发生。
\end{frame}

\section{另一条路线：binlog $\to$ MQ (CDC)}
% ============================================================

\begin{frame}{应用层 Outbox vs 数据库 CDC：本质区别}
\begin{tikzpicture}[node distance=8mm and 10mm,font=\scriptsize,
    every node/.append style={minimum width=22mm,minimum height=8mm}]
  \node[node-box,node-ok]              (app)  {应用层 Outbox\\gomall 现状};
  \node[node-box,below=of app,node-warn](cdc) {数据库 CDC\\binlog $\to$ MQ};
  \node[node-box,right=8mm of app]     (a1)   {业务代码 INSERT outbox};
  \node[node-box,right=of a1]          (a2)   {Publisher 轮询表};
  \node[node-box,right=of a2]          (a3)   {RMQ};
  \node[node-box,right=8mm of cdc]     (b1)   {业务代码只 INSERT order};
  \node[node-box,right=of b1]          (b2)   {Canal / Debezium 拉 binlog};
  \node[node-box,right=of b2]          (b3)   {Kafka / RMQ};
  \draw[arrow] (a1) -- (a2); \draw[arrow] (a2) -- (a3);
  \draw[arrow] (b1) -- (b2); \draw[arrow] (b2) -- (b3);
\end{tikzpicture}
\vspace{2mm}
\small \textbf{核心区别}：
\begin{itemize}
  \item Outbox 让业务\textbf{显式}声明"我要发什么事件"，Payload 是业务语义。
  \item CDC 让业务\textbf{完全无感}，事件由 binlog 行级变更倒推出来，Payload 是行差。
\end{itemize}
\srcnote{对照 service/order.go 与假设的 binlog 消费链路}
\end{frame}

\begin{frame}{Canal / Debezium / Maxwell 三家对比}
\begin{tabular}{@{}lp{2.8cm}p{2.8cm}p{2.8cm}@{}}
\toprule
工具 & Canal & Debezium & Maxwell \\
\midrule
出身       & 阿里开源 (Java)      & RedHat 开源 (Java)    & Zendesk 开源 (Java) \\
依赖       & 自带 server / client & Kafka Connect 框架    & 直接吐到 Kafka/Redis \\
DDL 支持   & 解析有限             & 完整                  & 完整 \\
位点存储   & ZK / 本地文件        & Kafka topic           & MySQL 表 / 文件 \\
适用场景   & 国内栈、ZK 已有       & 云原生、K8s、多源     & 中小项目快速接 \\
gomall 评估 & 运维有 ZK 才考虑    & K8s 成熟后再上        & 单体阶段过重 \\
\bottomrule
\end{tabular}
\vspace{2mm}
\small 三家都装成 MySQL 从库读 binlog，差异在\textbf{部署模型}和\textbf{下游协议}。gomall 现阶段不引入任何一家 —— Outbox 一张表 + 一个 goroutine 足够。
\srcnote{docs/blog/04-outbox-saga.md \S 业界路线 / 何时该切 CDC}
\end{frame}

\begin{frame}{什么时候 Outbox 该让位给 CDC}
\textbf{切换信号}（满足 2 个以上就该评估）：
\begin{enumerate}
  \item 业务表数量 $> 20$，每张表都要写 outbox 改造成本太高。
  \item 跨语言/跨团队消费者，需要"行级变更"这种通用契约。
  \item 历史数据回放 / 全量重建索引是常态需求。
  \item Publisher 已经横向扩展到 $\geq 4$ 实例还跟不上写入速度。
  \item 离线数仓 / OLAP 也要订阅同一份变更（CDC 天然支持 fanout）。
\end{enumerate}
\vspace{2mm}
\textbf{gomall 现状}：表 $\approx 12$、消费者全 Go、写入峰值远小于单 publisher 上限 —— \textbf{暂不切换}。
\keypoint{Outbox 的退场不是被淘汰，而是\textbf{规模换路线}。}
\srcnote{service/outbox/publisher.go 单实例 BatchSize=100/1s，无瓶颈}
\end{frame}

\section{Publisher：轮询 + 状态机推进}
% ============================================================

\begin{frame}{Publisher 默认参数：业务为什么这么取}
\begin{tabular}{@{}llp{4.8cm}@{}}
\toprule
参数 & 默认值 & 业务依据 \\
\midrule
\texttt{PollInterval} & \textbf{1s}    & 用户感知"延迟"上限 1s 是搜索/对账接受的窗口 \\
\texttt{BatchSize}    & \textbf{100}   & 单实例 + 1 张表 + 长事务规避，100 行单次能在 50ms 内 commit 完 \\
\texttt{MaxAttempts}  & \textbf{10}    & 指数退避 1s/2s/4s/.../上限 5min，10 次累计可覆盖约 17 分钟波动 \\
\bottomrule
\end{tabular}
\vspace{2mm}
\small 这三个值不是拍脑袋：\texttt{BatchSize=100} 对应"一次 \texttt{FetchBatch} 不会让 publisher 自己变成慢查询"；\texttt{MaxAttempts=10} 之后进 dead，由值班看 alarm。
\srcnote{service/outbox/publisher.go:14--32 \texttt{withDefaults}；repository/db/model/outbox.go:14 \texttt{OutboxDefaultMaxAttempts}}
\end{frame}

\begin{frame}[fragile]{publisher.loop / drainBatch：主循环 + 状态推进}
\begin{lstlisting}[language=Go,firstnumber=51,
  caption={\srcnote{service/outbox/publisher.go:51--86}}]
func (p *Publisher) loop(ctx context.Context) {
    ticker := time.NewTicker(p.cfg.PollInterval); defer ticker.Stop()
    for {
        select {
        case <-ctx.Done(): return
        case <-ticker.C:   p.drainBatch(ctx)
        }
    }
}
func (p *Publisher) drainBatch(ctx context.Context) {
    if rabbitmq.GlobalRabbitMQ == nil { return }
    rows, err := dao.NewOutboxDao(ctx).FetchBatch(p.cfg.BatchSize)
    if err != nil { util.LogrusObj.Errorln(err); return }
    for _, ev := range rows {
        if err := rabbitmq.PublishDomainEvent(ctx, ev.RoutingKey, []byte(ev.Payload)); err != nil {
            dao.NewOutboxDao(ctx).MarkFailed(ev.ID, ev.Attempts, p.cfg.MaxAttempts, err.Error())
            continue
        }
        dao.NewOutboxDao(ctx).MarkSent(ev.ID)
    }
}
\end{lstlisting}

func (d *OutboxDao) MarkFailed(id uint, attempts, maxAttempts int, errMsg string) error {
    now := time.Now()
    updates := map[string]any{
        "attempts":   attempts + 1,
        "last_error": truncate(errMsg, 500),
        "updated_at": now,
    }
    if attempts+1 >= maxAttempts {
        updates["status"] = model.OutboxStatusDead
    } else {
        backoff := time.Duration(1<<attempts) * time.Second
        if backoff > 5*time.Minute { backoff = 5 * time.Minute }
        updates["next_retry_at"] = now.Add(backoff)
    }
    return d.DB.Model(&model.OutboxEvent{}).Where("id = ?", id).Updates(updates).Error
}
\end{lstlisting}
\vspace{1mm}
\small 逐行讲（核心 4 点）：\texttt{1 << attempts} 实现退避 1s/32s/256s，被 5min 封顶；退避结果写 \texttt{next\_retry\_at} 让下次 FetchBatch 自动过滤；\texttt{attempts+1 >= max} 第 10 次失败才进 dead；\texttt{last\_error} 截断 500 字符防 LOB 撑爆。

\begin{frame}{死信处理：进 dead 状态之后该做什么}
\begin{itemize}
  \item dead 状态\textbf{不会自愈} —— 这是设计选择：让人介入比让代码盲目重试更安全。
  \item \textbf{监控指标}（Prometheus）：
  \begin{itemize}
    \item \texttt{outbox\_pending\_total}：堆积量，超过 $10k$ 报警。
    \item \texttt{outbox\_dead\_total}：累计死信，每出现一次就报警。
    \item \texttt{outbox\_publish\_latency\_p99}：单条 publish 延迟，p99 $> 1s$ 报警。
  \end{itemize}
  \item \textbf{人工处置}：值班看 \texttt{last\_error}，确认下游修复后用 SQL 改回 \texttt{status=1, attempts=0, next\_retry\_at=now()}。
  \item \textbf{反例}：自动把 dead 改回 pending —— 等于把"无限重试"绕过 MaxAttempts，下游一旦挂掉就被打死。
\end{itemize}
\keypoint{死信信箱的作用是让\textbf{失败可见}，不是让\textbf{失败被自动忽略}。}

// TryProductLock SETNX 抢回源锁，避免缓存击穿时多个请求同时回源
func TryProductLock(ctx context.Context, id uint) (bool, error) {
    return RedisClient.SetNX(ctx, ProductLockKey(id), "1", ProductLockTTL).Result()
}

func UnlockProduct(ctx context.Context, id uint) {
    RedisClient.Del(ctx, ProductLockKey(id))
}

// DoubleDeleteAsync 延迟双删：写库后已经做了第一次删除，这里在 interval 后再删一次，
// 用于覆盖"读旧值的并发请求把旧值塞回缓存"的窗口。
func DoubleDeleteAsync(id uint, interval time.Duration) {
    if interval <= 0 { interval = ProductDelayInterval }  // 默认 500ms
    go func() {
        time.Sleep(interval)
        _ = RedisClient.Del(context.Background(), ProductDetailKey(id)).Err()
    }()
}
\end{lstlisting}

\section{缓存一致性四模式：业务侧选型对比}
% ============================================================

\begin{frame}{Cache Aside / Write Through / Write Back / Read Through}
\begin{tabular}{@{}lp{3.2cm}p{3.0cm}p{2.6cm}@{}}
\toprule
模式 & 写路径 & 读路径 & 典型问题 \\
\midrule
Cache Aside     & 业务写 DB $\to$ 删 cache  & cache miss $\to$ 读 DB $\to$ 回填 & 双删窗口 (gomall) \\
Write Through   & 业务写 cache $\to$ cache 同步写 DB & 直接读 cache  & cache 即真相，故障重 \\
Write Back      & 业务写 cache，cache 异步刷 DB     & 直接读 cache  & cache 宕机丢数据 \\
Read Through    & 业务只跟 cache 说话；miss 时 cache 自己回源 & 同左 & 需要 cache 中间层 \\
\bottomrule
\end{tabular}
\vspace{2mm}
\small \textbf{gomall 选 Cache Aside}：DB 是真相，cache 失败不影响业务正确性。Write Back 虽快（写 cache 1ms / 异步落 DB），但 Redis 宕机直接丢数据、对账困难、冷启动慢 —— 订单/库存\textbf{绝不}走 Write Back；只有"商品浏览次数"这种允许 1\% 丢失的副表才用。\keypoint{选模式的口诀：\textbf{写延迟 $\leftrightarrow$ 容忍丢失}的折中。}
\srcnote{repository/cache/product.go GetProductDetail / SetProductDetail / DelProductDetail}
\end{frame}

\section{启动顺序事故：InitLog 必须在 RMQ 之前}
% ============================================================

\begin{frame}{真实事故：\texttt{tryInitRabbitMQ} 让进程整个退出}
\textbf{现象}：RMQ 没起，\texttt{tryInitRabbitMQ()} \texttt{recover} 时 \textbf{NPE}，进程 crash。

\vspace{2mm}
\textbf{定位过程}：
\begin{enumerate}
  \item PR \#51（订单延迟关单）引入 \texttt{tryInitRabbitMQ()}。
  \item 该函数 \texttt{recover()} 内调用 \texttt{util.LogrusObj.Warnf(...)}。
  \item 当时 \texttt{util.InitLog()} 写在 \texttt{tryInitRabbitMQ()} \textbf{之后}。
  \item \texttt{LogrusObj == nil} $\to$ recover 内再 panic $\to$ 进程退出。
\end{enumerate}
\vspace{1mm}
\textbf{修复（commit 8d0157a / 3cbe3c4）}：把 \texttt{util.InitLog()} 提前到所有可能 \texttt{recover} 的代码之前。
\srcnote{cmd/main.go:30--31 修复后的顺序；stressTest/REPORT.md \S 已修：util.LogrusObj}
\end{frame}

\begin{frame}{两条可复用的业务侧教训}
\begin{enumerate}
  \item \textbf{初始化顺序}：所有依赖全局对象的代码，必须在它初始化之后才能调用。
  \begin{itemize}
    \item \texttt{InitLog} / \texttt{InitTracer} / \texttt{InitMetrics} 放最前。
    \item 用 \texttt{init()} 隐式引导时盯紧 import 顺序。
  \end{itemize}
  \item \textbf{recover 内禁止再 panic}：
  \begin{itemize}
    \item recover 是最后兜底，里面再崩进程彻底失控。
    \item 写法：recover 内只 \texttt{fmt.Fprintln(os.Stderr, ...)} 或写本地文件，不依赖任何复杂组件。
  \end{itemize}
\end{enumerate}
\keypoint{outbox publisher 起不来不算事故 —— 启动顺序错了让进程 crash，才是事故。}
\srcnote{cmd/main.go:54--62 tryInitRabbitMQ 的 recover 块；commit 8d0157a 修复点}
\end{frame}

\section{真实事故复盘：一致性怎么在生产环境炸}
% ============================================================

\begin{frame}{事故一：未支付订单取消导致虚高库存}
\textbf{现象}（commit 5c9ae27 前）：压测连续下单+不支付，15 分钟后\textbf{商品可售数变成负数}。

\vspace{2mm}
\textbf{根因链}：
\begin{enumerate}
  \item 旧版 \texttt{CancelUnpaidOrder} 把 \texttt{product.num} 加回 DB —— 但未支付订单\textbf{从未真正扣减过} DB 库存 (只占了 Redis reserved)。
  \item 第一次取消：DB.num \texttt{+= n}；Redis.reserved \texttt{-= n} —— 库存虚高 n。
  \item 累计 100 个超时订单 = 库存虚高 100 倍。
\end{enumerate}
\vspace{1mm}
\textbf{修复}（PR \#58 引入预扣 + Saga）：
\begin{itemize}
  \item DB.num 在\textbf{支付成功}时才递减，未支付订单只动 Redis reserved。
  \item CancelUnpaidOrder \textbf{不再}写 \texttt{product.num}，只释放 Redis reserved。
  \item 库存不变量：\texttt{available + reserved + sold = total} 始终成立。
\end{itemize}
\srcnote{service/order\_cancel.go:14--63 注释 "不再回写 product.Num"}
\end{frame}

\begin{frame}{事故二：搜索索引腐烂 12 小时无人知}
\textbf{现象}（commit a43cf61 前）：商品改价后，搜索结果一直显示旧价；商详页正常。
\vspace{2mm}

\textbf{根因链}：
\begin{enumerate}
  \item 旧链路：商品 UPDATE 时\textbf{同步}调用 ES 重建索引。
  \item ES 集群偶发拒连 $\to$ 同步调用失败 $\to$ 业务 \texttt{rollback}。
  \item 业务为了不影响下单，把 ES 写入改成\textbf{忽略错误}。
  \item ES 失败被吞，索引与 DB 漂移 12 小时无人知。
\end{enumerate}
\vspace{1mm}
\textbf{修复}（PR \#59 切到 outbox + 消费者）：
\begin{itemize}
  \item 商品 UPDATE 同事务\textbf{只写 outbox\_event}，不直接写 ES。
  \item ES 消费者从 RMQ 拉事件，失败有重试 + 死信。
  \item 死信触发 alarm，值班同学 5 分钟内看到。
\end{itemize}
\srcnote{commit a43cf61 商品搜索切换到 ES 通过 outbox + 消费者保持增量索引}
\end{frame}

\begin{frame}{事故三：进程整体 crash 而非降级}
\textbf{现象}：RabbitMQ 服务下线维护时，gomall 进程\textbf{整体退出}，HTTP 接口全 503。

\vspace{2mm}
\textbf{根因链}：
\begin{enumerate}
  \item \texttt{tryInitRabbitMQ()} 的 \texttt{recover} 块里调用 \texttt{LogrusObj.Warnf}。
  \item 初始化序列中 \texttt{InitLog()} 反而在它\textbf{之后}。
  \item \texttt{LogrusObj == nil} $\to$ recover 内再次 panic $\to$ 进程退出。
  \item 业务影响：RMQ 是\textbf{可选依赖}（延迟关单可降级），但实际把整个进程拖死。
\end{enumerate}
\vspace{1mm}
\textbf{修复}（commit 8d0157a / 3cbe3c4）：
\begin{itemize}
  \item \texttt{InitLog()} 上调到 \texttt{InitConfig()} 之后第一个位置。
  \item \texttt{recover} 内禁止调用任何依赖全局对象的代码。
  \item 把 RMQ / ES / Web3 / Milvus 全部走 \texttt{tryInit*} 包装，降级而非崩溃。
\end{itemize}
\srcnote{cmd/main.go:30--47 / stressTest/REPORT.md \S 已修：util.LogrusObj}
\end{frame}

\begin{frame}{面试 Q\&A（1/3）}
\textbf{Q1}：为什么 Outbox 比"DB commit 后再 publish"安全？\\
\hspace{4mm}\small A：commit 与 publish 不在一个事务，commit 成功 publish 失败就丢事件。Outbox 把事件写进 DB，由轮询消化，事件丢失等价于 DB 数据丢失。\srcnote{service/order\_cancel.go:30--51}

\vspace{2mm}
\textbf{Q2}：outbox 表会不会堆积？\\
\hspace{4mm}\small A：会。监控 \texttt{status=pending AND next\_retry\_at <= now()} 的行数即可。横向扩展可用 \texttt{SELECT FOR UPDATE SKIP LOCKED}；当前 gomall 单实例 BatchSize=100/1s 还远没到瓶颈。\srcnote{repository/db/dao/outbox.go:44--51}

\vspace{2mm}
\textbf{Q3}：延迟双删的"延迟"该取多大？\\
\hspace{4mm}\small A：取\textbf{大于}最长读 race 窗口。gomall 取 500ms：DB 写常态 5--20ms，留 20$\times$ 余量盖 GC / 网络抖动；TTL 10min 兜底。\srcnote{repository/cache/product.go:64--74}

\vspace{2mm}
\textbf{Q4}：协同式 vs 编排式 Saga，怎么选？\\
\hspace{4mm}\small A：步骤 $\leq 5$、跨团队少 $\to$ 协同（gomall 选）。步骤多、需可视化、跨团队多 $\to$ 编排，但要承担"中心节点"运维成本。\srcnote{docs/blog/04-outbox-saga.md \S 完整链路}
\end{frame}

\begin{frame}{面试 Q\&A（2/3）}
\textbf{Q5}：为什么 cache 释放失败不回滚 DB？\\
\hspace{4mm}\small A：DB 是业务真相，cache 是性能侧。Cache 失败让库存"短暂少一点"，最坏拒单不超卖；回滚 DB 反而让"已关单"被复活，业务码乱套。\srcnote{service/order\_cancel.go:58--62}

\vspace{2mm}
\textbf{Q6}：为什么不用 CDC (Debezium / binlog) 替代 Outbox？\\
\hspace{4mm}\small A：CDC 更通用但要拉 binlog 链路、跑 Debezium 集群，运维重；Outbox 是"应用层 CDC"，简单可控，gomall 量级足够。\srcnote{service/outbox/publisher.go 全局}

\vspace{2mm}
\textbf{Q7}：2PC / XA 为什么互联网业务不用？\\
\hspace{4mm}\small A：协调者单点 + 同步阻塞 + 异构资源不支持 (Redis/RMQ/ES 都没 XA)。CAP 三选二里 2PC 强 CP，gomall 选 AP + 业务码不变量。\srcnote{docs/blog/04-outbox-saga.md \S 为什么不用 2PC}

\vspace{2mm}
\textbf{Q8}：TCC 与 Outbox 在 gomall 里有什么映射？\\
\hspace{4mm}\small A：库存 reserved 字段实际上就是 TCC 的 Try 阶段，但 Confirm/Cancel 不走 TCC 同步协议，改成 outbox 异步事件触发 —— 等价语义、运维低一档。\srcnote{service/inventory/reservation.go}
\end{frame}

\begin{frame}{面试 Q\&A（3/3）}
\textbf{Q9}：MaxAttempts=10 怎么算出来的？退避 5min 封顶又是为什么？\\
\hspace{4mm}\small A：1+2+4+8+16+32+64+128+256+300=811s$\approx$13.5min 累计窗口，覆盖一次"中等故障"够用；封顶 5min 防止退避无限拉长导致积压。\srcnote{repository/db/dao/outbox.go:62--78}

\vspace{2mm}
\textbf{Q10}：Outbox 表会不会成为整库的写入热点？\\
\hspace{4mm}\small A：当前峰值 $< 100$ ev/s，瓶颈在 publish 而非 INSERT；如果突破 \texttt{innodb\_io\_capacity}，按 \texttt{aggregate\_type} 分表 + 不同 publisher 消费。\srcnote{repository/db/model/outbox.go idx\_outbox\_dispatch 索引}

\vspace{2mm}
\textbf{Q11}：Event Sourcing 比 Outbox 强在哪？为什么 gomall 不上？\\
\hspace{4mm}\small A：ES 把事件流当真相，天然支持回放/审计；代价是读模型必须维护投影 + 快照。gomall 当前规模 ROI 不正，但 outbox\_event 表只追加、不删除，已经为未来演进留了门。\srcnote{repository/db/dao/outbox.go 无 Delete 路径}

\vspace{2mm}
\textbf{Q12}：业务侧怎么承诺"最终一致"的窗口？\\
\hspace{4mm}\small A：分三档：库存预扣 100ms、订单/搜索可见 1s、跨服务对账 5min。对外 SLO 写明上限，超过触发 P2 报警。\srcnote{service/outbox/publisher.go PollInterval=1s}
\end{frame}
