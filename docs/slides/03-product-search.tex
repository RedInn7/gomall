\begin{frame}{业务对搜索的四条硬约定}
\begin{itemize}
  \item \textbf{召回}：用户搜得到的词，能跑出至少一条相关结果。
  \item \textbf{时延}：p95 < 200ms，列表页才不会"卡半秒空白"。
  \item \textbf{新鲜度}：商家上架 $\to$ 站内可搜，SLA $\le$ 5min。
  \item \textbf{兜底}：ES 挂掉，搜索接口不能 5xx，必须有降级路径。
\end{itemize}
\vspace{2mm}
\keypoint{这四条决定了搜索系统不能只是 \texttt{SELECT * WHERE name LIKE '\%kw\%'}。}
\srcnote{stressTest/REPORT.md：/product/show 同库 PK 查询基线 62{,}226 RPS / p95 3ms 作为时延锚点}
\end{frame}

\begin{frame}{字段加权背后的业务判断}
\begin{tabular}{@{}llr@{}}
\toprule
字段          & 业务含义                       & 加权     \\
\midrule
\texttt{name}  & 商品主名（用户预期匹配）        & $\times 3$ \\
\texttt{title} & 商品标题（含型号 / 规格）       & $\times 2$ \\
\texttt{info}  & 商品详情（长文，噪声多）        & $\times 1$ \\
\bottomrule
\end{tabular}
\vspace{2mm}
\begin{itemize}
  \item 用户搜 "iPhone"，"标题就叫 iPhone 13" 应当排在"info 提到 iPhone"前面。
  \item 加权由运营调，不是开发拍。大促可临时把 \texttt{title} 上调到 $\times 4$。
  \item 不加权 $\to$ "标签店家把热词塞进 info" 会霸占第一屏。
\end{itemize}

\end{frame}

\begin{frame}{代码段 1 逐行讲解}
\begin{itemize}
  \item \textbf{L170--175}：\texttt{multi\_match} 同时查 name / title / info，靠 \texttt{\^{}N} 给字段加权。这一行就是上一页那张加权表的代码落点。
  \item \textbf{L176--180}：\texttt{filter} 走 \texttt{term}，类目过滤不参与打分，只做 yes/no 过滤，性能更稳。运营按类目限定时进这条分支。
  \item \textbf{L182--189}：\texttt{bool.must + filter} 是 ES 的标准组合 —— must 决定相关性，filter 决定可见性。两者分离，分页和打分才不会互相干扰。
  \item \textbf{业务对应}：name 权重最高，对应"用户最在意主名命中"；filter 不打分，对应"类目是硬过滤，不是建议"。
\end{itemize}
\end{frame}

\begin{frame}{must / should / filter / must\_not 四件套}
\begin{tabular}{@{}lll@{}}
\toprule
子句         & 是否打分 & 典型用途                       \\
\midrule
\texttt{must}    & 是      & 关键词召回，决定相关性          \\
\texttt{should}  & 是      & 可选加分项，命中即上调          \\
\texttt{filter}  & 否      & 硬过滤：类目 / 上架 / 库存 \textgreater 0 \\
\texttt{must\_not} & 否      & 黑名单：下架 / 违禁 / 自家店     \\
\bottomrule
\end{tabular}
\vspace{2mm}
\begin{itemize}
  \item 打分子句进 query cache miss 时重算，filter 子句进 filter cache 长期命中。
  \item 把所有"硬条件"塞进 filter 是最常见的性能优化，可使 QPS 翻倍。
  \item \texttt{should} 配合 \texttt{minimum\_should\_match} 还能做"至少命中 N 个同义词"。
\end{itemize}
\srcnote{ES 官方 query DSL 与 gomall 用法已对齐：repository/es/product\_index.go:170--191}
\end{frame}

\begin{frame}{BM25 vs TF-IDF：现代 ES 默认值}
\begin{itemize}
  \item TF-IDF：\texttt{tf} 直接线性，长文档天然占便宜。
  \item BM25：\texttt{tf} 走饱和函数 \texttt{(k+1)*tf / (k+tf*\dots)}，词频再高也不会无脑加分。
  \item BM25 引入文档长度归一 \texttt{b}，长 info 不再压制短 name。
  \item ES 7.x 起默认 \texttt{similarity=BM25}，gomall 不显式覆盖，保留默认。
\end{itemize}
\keypoint{BM25 是"被工业界打磨过 20 年的 TF-IDF"；运营提"搜索结果偏长描述"时，先验证是不是误改了 \texttt{k1/b}。}
\srcnote{repository/es/product\_index.go:20--41 mapping 未覆盖 similarity，沿用集群默认 BM25}
\end{frame}

\section{上架到可搜：outbox + 异步索引 + 兜底}
% ============================================================

\begin{frame}{业务承诺：商家上架 5 分钟内必须搜得到}
\begin{itemize}
  \item 商家后台点"上架" $\to$ 站内搜索能搜到，业务侧的 SLA 是 $\le$ 5min。
  \item 5min 是商家容忍上限：拍照 / 调价 / 改文案的反馈节奏。
  \item 写 DB 同步写 ES 行不行？\textbf{不行}：ES 抖动直接 fail 上架，商家退出。
  \item 必须\textbf{解耦}：DB 写完就返回成功，ES 异步追上。
\end{itemize}
\srcnote{service/product.go:155, 253, 289 三处商品写入后调 emitProductChanged}
\end{frame}

\begin{frame}[fragile]{代码段 2：indexer 主循环}
\begin{lstlisting}[language=Go,firstnumber=17,basicstyle=\ttfamily\tiny]
func StartProductIndexer(ctx context.Context) error {
    rabbitmq.BindDomainQueue(indexerQueue, "product.changed")
    ch, _ := rabbitmq.GlobalRabbitMQ.Channel()
    ch.Qos(32, 0, false)  // 在途上限 32 条
    msgs, _ := ch.Consume(indexerQueue, "", false, false, false, false, nil)
    go func() {
        for d := range msgs {
            var ev events.ProductChanged
            if json.Unmarshal(d.Body, &ev) != nil {
                d.Nack(false, false); continue  // 脏消息进死信
            }
            if handleProductChanged(ctx, ev) != nil {
                d.Nack(false, true);  continue  // 重投自愈
            }
            d.Ack(false)
        }
    }()
    return nil
}
\end{lstlisting}
\srcnote{service/search/indexer.go:17--49（折行省略错误处理）}
\end{frame}

\begin{frame}{代码段 2 逐行讲解}
\begin{itemize}
  \item \textbf{L18--19}：\texttt{BindDomainQueue} 把 \texttt{product.changed} 路由绑到 \texttt{search.product.indexer} 队列。上架 / 改价 / 下架共用一条路由。
  \item \textbf{L22}：\texttt{Qos(32, 0, false)} 让消费端最多在途 32 条未 ack。这是给 ES 写入设的"流量阀"，对应 deck 05 的"在网关之外，每个消费者也要节流"。
  \item \textbf{L30--34}：JSON 解析失败 $\to$ \texttt{Nack(false, false)}，不重投。脏消息进死信，不阻塞健康消息。
  \item \textbf{L35--38}：业务处理失败 $\to$ \texttt{Nack(false, true)} 重投，给 ES 短抖动一次自愈机会。
  \item \textbf{L39}：成功才 ack，保证 at-least-once。索引是幂等的（同 docID 覆盖），重复消费无害。
\end{itemize}
\srcnote{service/search/indexer.go:25 Qos=32；:36/:42 Nack 语义}
\end{frame}

\begin{frame}{代码段 3 逐行讲解}
\begin{itemize}
  \item \textbf{shards=1 / replicas=0}：gomall 单节点部署，不做多副本。生产换多节点要把 replicas 拉到 1，read 走主+副两副本。
  \item \textbf{name/title/info 是 \texttt{text}}：参与分词 + 倒排，是关键词召回的目标字段。
  \item \textbf{price 是 \texttt{keyword}}：不分词，整串做 filter / exact match。价格永远不会被"分词"成"99" + "元"。

\begin{frame}{面试 Q\&A (1/3)}
\textbf{Q1}：为什么不让上架 API 直接写 ES，而要 outbox？\\
\hspace{4mm}\small A：DB 与 ES 不同事务，同步写要么失败回滚（伤体验），要么不一致（伤召回）。outbox 把"DB 写成功" 和 "ES 写成功" 两件事拆开，分别保证。代码：\srcnote{service/product.go:257--265}

\vspace{2mm}
\textbf{Q2}：ES 挂掉，搜索接口为什么不直接 5xx？\\
\hspace{4mm}\small A：搜索是入口，5xx 等于 app 崩页。LIKE 慢一点也比白屏强 —— 业务承诺"搜索框永远可用"。代码：\srcnote{service/product.go:294--323}

\vspace{2mm}
\textbf{Q3}：hybrid 50/50 这个权重你怎么调？\\
\hspace{4mm}\small A：业务值，不是技术值。先 50/50 跑一轮，A/B 观察 CTR / 转化。短 query 偏关键词，长 query 偏语义。代码：\srcnote{service/search/semantic.go:21--23}

\vspace{2mm}
\textbf{Q4}：BM25 和 TF-IDF 在生产里有什么实操差别？\\
\hspace{4mm}\small A：BM25 词频走饱和函数，长文档不再天然占便宜；ES 7.x 起默认 BM25。运营说"排序偏长描述"先查 \texttt{k1/b} 是否被改。
\end{frame}

\begin{frame}{面试 Q\&A (2/3)}
\textbf{Q5}：min-max 归一在什么场景会失效？\\
\hspace{4mm}\small A：所有分数等值时 span=0。代码里返回常量 1，意味着"两路一样可信"，让另一路决定排序。代码：\srcnote{service/search/semantic.go:186--192}

\vspace{2mm}
\textbf{Q6}：HNSW 的 efSearch 你怎么调？\\
\hspace{4mm}\small A：efSearch 是查询时旋钮，不用重建索引。延迟告警时调小（64$\to$32）召回率损 1\%--3\%，P95 减半。代码：\srcnote{repository/milvus/product\_vector.go:141}

\vspace{2mm}
\textbf{Q7}：Milvus 挂了怎么办？为什么不直接 5xx？\\
\hspace{4mm}\small A：nopMilvusSearcher 返回空，hybrid 退化为纯 ES。关键词召回不受影响，只丢自然语言长尾。代码：\srcnote{service/search/milvus\_stub.go:19--24}

\vspace{2mm}
\textbf{Q8}：换 embedding 模型要走什么流程？\\
\hspace{4mm}\small A：维度必须仍是 768（mapping 硬约束），先 staging 跑 NDCG/MRR 评估，再灰度。代码：\srcnote{service/search/embedding.go:17--24}
\end{frame}

\begin{frame}{面试 Q\&A (3/3) + 代码位置}
\textbf{Q9}：5 分钟新鲜度 SLA 在哪一步可能掉？\\
\hspace{4mm}\small A：堆积一般出现在 RMQ 与 indexer 之间，Qos=32 + ES refresh\_interval=1s 下 p95 < 30s。监控 \texttt{search.product.indexer} queue\_depth。代码：\srcnote{service/search/indexer.go:25}

\vspace{2mm}
\textbf{Q10}：100M SKU 时这个架构还能用吗？\\
\hspace{4mm}\small A：ES 切多分片 + 副本；HNSW 内存撑不住改 IVF\_PQ 量化；冷热分层把老 SKU 放冷索引。\srcnote{repository/milvus/product\_vector.go:80 NewIndexHNSW}

\vspace{2mm}
\textbf{Q11}：以图搜图怎么接？\\
\hspace{4mm}\small A：CLIP/SigLIP 编码图片到 512/768 维向量，单独 collection \texttt{image\_vector}，走相同 outbox 路由。\srcnote{规划中：repository/milvus/ 新增 image\_vector}

\vspace{2mm}
\textbf{Q12}：mapping 改了字段类型怎么办？\\
\hspace{4mm}\small A：ES mapping 字段类型不能原地改，必须 \texttt{reindex} 或新建索引 + alias 切换。回填走 admin/search/backfill。\srcnote{routes/router.go:163}
\end{frame}
