\section{商家数据现状：BossID 在、接口缺}
% ============================================================

\begin{frame}{一个商家在 gomall 里能干什么、不能干什么}
\begin{itemize}
  \item \textbf{能干}：注册账号、上架商品、被买家下单、被买家收藏。
  \item \textbf{不能干}：查"我的今日营收"、查"我的待发货"、查"我的库存告警"、自助退款。
  \item 五张核心表里 \textbf{BossID 字段早就埋好了}：\texttt{order.BossID}、\texttt{product.BossID}、\texttt{cart.BossID}、\texttt{favorite.BossID}、\texttt{skill\_product.BossID}。
  \item 但是 \texttt{/api/v1/merchant/*} \textbf{一个路由都没有} —— 数据有、查询入口缺。
\end{itemize}
\srcnote{repository/db/model/order.go:12、product.go:23、cart.go:10、favorite.go:12 五处 BossID}
\end{frame}

\begin{frame}{现状对业务的真实伤害}
\begin{itemize}
  \item \textbf{招商场景}：BD 谈一个新品牌，对方第一句话"我能看到自己卖了多少吗" $\to$ 答不上来。
  \item \textbf{客服场景}：商家把客服当后台用 —— "帮我查下我店铺今日订单"、"帮我看 SKU 12345 库存"。1 个客服对 N 个商家，工单全是低价值查询。
  \item \textbf{越权风险}：临时给商家开 admin 账号 $\to$ 商家立刻看到\textbf{所有店铺}数据，谈判桌上当场翻车。
  \item \textbf{数据正确但权限模型没成形}：\texttt{product/delete} 已经按 \texttt{WHERE id=? AND boss\_id=?} 隔离，但没有任何中间件保证调用者就是 \texttt{boss\_id} 这个人。
\end{itemize}
\srcnote{repository/db/dao/product.go:67--72 已带 boss\_id 隔离；routes/router.go:148--155 admin 子组只有 3 个 handler}
\end{frame}

\begin{frame}[fragile]{RequireRole + lookupRole：30 秒缓存的取舍}
\begin{lstlisting}[language=Go,firstnumber=22,
  caption={\srcnote{middleware/rbac.go:22--45 + :89--91}}]
var (
    roleCache    sync.Map
    roleCacheTTL = 30 * time.Second
)
func lookupRole(ctx context.Context, userId uint) (string, error) {
    if v, ok := roleCache.Load(userId); ok {
        e := v.(roleCacheEntry)
        if time.Now().Before(e.expires) { return e.role, nil }
    }
    u, err := dao.NewUserDao(ctx).GetUserById(userId)
    if err != nil { return "", err }
    role := u.Role
    if role == "" { role = "user" }
    roleCache.Store(userId, roleCacheEntry{role: role,
        expires: time.Now().Add(roleCacheTTL)})
    return role, nil
}
func InvalidateRoleCache(id uint) { roleCache.Delete(id) }
\end{lstlisting}
\end{frame}

\begin{frame}{lookupRole 逐行讲解}
\begin{itemize}
  \item \textbf{为什么要缓存}：每个 admin / merchant 请求都进 \texttt{RequireRole}，不缓存就每次打 DB；高并发下 user 表的 \texttt{SELECT WHERE id=?} 会变成路由瓶颈。
  \item \textbf{为什么是 30s}：商家提权 / 降级是低频事件，30s 内权限漂移可接受；同时单点回源 DB 的 QPS 被砍到 1/30。
  \item \textbf{为什么用 sync.Map}：单实例进程内只读为主、写少；不引入 Redis 避免把鉴权链路绑到 Redis 健康度上。
  \item \textbf{为什么提权后必须显式 Invalidate}：商家刚被审核通过，第一次访问 \texttt{/merchant/*} 不能等 30s。\texttt{service/admin.go:59} 调 \texttt{InvalidateRoleCache(targetUserId)} 做精确失效。
\end{itemize}
\srcnote{service/admin.go:59 PromoteToAdmin 末尾的显式失效}
\end{frame}

\section{Jaeger：链路追踪解决的真实问题}
% ============================================================

\begin{frame}{为什么要在 router 第一行就挂 Jaeger}
\begin{itemize}
  \item gomall 一次下单要穿过：\texttt{ratelimit} $\to$ \texttt{Jaeger} $\to$ \texttt{Cors} $\to$ \texttt{Auth} $\to$ \texttt{Idempotency} $\to$ \texttt{OrderCreate} $\to$ \texttt{Redis Lua} $\to$ \texttt{MySQL} $\to$ \texttt{Outbox}。
  \item 出问题时\textbf{只看错误日志}永远定位不到瓶颈：哪一段慢、哪一段挂、哪一段触发了重试 —— 日志里只有"成功"或"失败"。
  \item Jaeger 把这条链做成一棵 \textbf{span 树}：每段耗时、每次依赖调用、每个上下游 traceId 都能串起来。
  \item 业务价值：客服报"用户 X 下单 5 秒才返回" $\to$ 用 trace 一找 $\to$ 看到 4.7s 卡在第三方支付的 \texttt{paydown} 步。
\end{itemize}
\srcnote{routes/router.go:23 全局挂 \texttt{middleware.Jaeger()}}
\end{frame}

\begin{frame}[fragile]{Jaeger middleware：traceId 透传}
\begin{lstlisting}[language=Go,firstnumber=11,
  caption={\srcnote{middleware/track.go:11--29}}]
func Jaeger() gin.HandlerFunc {
    return func(c *gin.Context) {
        traceId := c.GetHeader("uber-trace-id")
        var span opentracing.Span
        if traceId != "" {
            var err error
            span, err = track.GetParentSpan(
                c.FullPath(), traceId, c.Request.Header)
            if err != nil { return }
        } else {
            span = track.StartSpan(
                opentracing.GlobalTracer(), c.FullPath())
        }
        defer span.Finish()
        c.Set(consts.SpanCTX, opentracing.ContextWithSpan(c, span))
        c.Next()
    }
}
\end{lstlisting}
\end{frame}

\begin{frame}{Jaeger middleware 逐行讲解}
\begin{itemize}
  \item \textbf{第 3 行}：读 \texttt{uber-trace-id} header；格式 \texttt{<traceId>:<spanId>:<parentId>:<flags>}。
  \item \textbf{第 5--9 行}：有 traceId $\to$ 当前请求是别人调过来的，开 \textbf{child span}，root 留在上游。
  \item \textbf{第 10--13 行}：没 traceId $\to$ 当前请求是入口，自己开 root span。
  \item \textbf{第 14 行}：\texttt{defer span.Finish()} 必须有，否则 Jaeger 端会看到"未结束"灰色 trace。
  \item \textbf{第 15 行}：把 span 写回 \texttt{c.Context}，下游通过 \texttt{ctl.GetSpanContext} 开 child span —— \textbf{这一步就是"链路"的串接}。
\end{itemize}
\srcnote{pkg/utils/track/track.go:61--78 GetParentSpan Extract}
\end{frame}

func tryInitES(ctx context.Context) {
    defer func() {
        if r := recover(); r != nil {
            util.LogrusObj.Warnf("ES 初始化失败，搜索退化: %v", r)
        }
    }()
    es.InitEs()
    initialize.InitSearch(ctx)
}
\end{lstlisting}

\begin{frame}{tryInitES 逐行讲解：为什么不传 error}
\begin{itemize}
  \item \textbf{第 3--8 行 defer + recover}：底层 \texttt{es.InitEs()} 是\textbf{老代码}，连不上 ES 直接 \texttt{panic}。改 \texttt{InitEs} 签名风险大（连锁影响 search service），所以\textbf{在外层包一个 recover}。
  \item \textbf{为什么不返回 error 让 main 决定}：main 拿到 error 也只能写日志接着启 —— 业务上"ES 挂"和"启动失败"是\textbf{两件事}，前者不应阻塞 HTTP 启动。
  \item \textbf{为什么用 Warnf 不是 Errorf}：error 级别会触发告警；这里是"已知降级路径"，不是异常。给 SRE 留 Warn 让他主动看，不要凌晨 3 点叫人起床。
  \item \textbf{相同套路用了 4 次}：\texttt{tryInitRabbitMQ}、\texttt{tryInitES}、\texttt{tryInitWeb3Listener}、\texttt{tryInitMilvus} —— 同一种姿势，业务上是\textbf{对"不可控外部依赖"的统一回应}。
\end{itemize}
\srcnote{cmd/main.go:54--97 四个 tryInit 同构}
\end{frame}

\begin{frame}{面试 Q\&A}
\textbf{Q1}：BossID 都在了，为什么不直接给商家开 admin？\\
\hspace{4mm}\small A：admin 看全站数据，商家只能看自店；权限边界不能用"先开放再监控"的姿态做。merchant 角色 + RequireRole("merchant") 才是正确解。

\vspace{2mm}
\textbf{Q2}：Jaeger 在 64K RPS 下还撑得住吗？\\
\hspace{4mm}\small A：实测 \texttt{/product/show} 全中间件跑 62{,}226 RPS（\texttt{/ping} 64{,}254），开销 $<$ 5\%。const 全量采样能压得住；担心损耗可切 probabilistic 0.001\%。

\vspace{2mm}
\textbf{Q3}：tryInitES 为什么用 recover 不传 error？\\
\hspace{4mm}\small A：底层 \texttt{es.InitEs()} 老代码 panic 风格；改签名牵连大。外层 recover + Warnf 是\textbf{对"启动期不可控依赖"的统一回应}，与请求期熔断（70002）形成两套互补。
\end{frame}

\begin{frame}{Q\&A（续）}
\textbf{Q4}：roleCache 为什么 30s 不是 5min？\\
\hspace{4mm}\small A：商家被提权后等 5min 才生效，谈判桌上没法解释。30s 是"权限漂移可接受 + 回源 DB 砍 30 倍"的平衡点；提权时显式 \texttt{InvalidateRoleCache} 让\textbf{当事人立即生效}。

\vspace{2mm}
\textbf{Q5}：Jaeger 和 Skywalking 同时存在，是不是重复建设？\\
\hspace{4mm}\small A：单服务调试 Jaeger 细，多服务拓扑 Skywalking 直观。两者数据模型相同（trace + span），并存只是双写不冲突；当 gomall 真上微服务后，Skywalking 的全局拓扑图是 SRE 的主视角。

\vspace{2mm}
\textbf{Q6}：商家自助接口和 admin 接口，业务码段为什么要分？\\
\hspace{4mm}\small A：商家看到的错误码会出现在\textbf{商家 App 文案}里；admin 看到的只在内部后台。两套话术系统不同，业务码隔离段能避免文案串号。
\end{frame}
