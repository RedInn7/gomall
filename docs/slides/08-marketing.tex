\section{优惠券：库存 + 限领 二维原子}
% ============================================================

\begin{frame}[fragile]{coupon Lua 脚本（业务画像 + 10 行实现）}
\small \texttt{coupon:batch:\{id\}:stock} 整型库存；\texttt{coupon:user:\{uid\}:batch:\{id\}} 单人计数 TTL 24h。Lua 必须一锤完成：库存 > 0 ？单人 < \texttt{PerUser} ？两道闸过了才 DECR + INCR。
\begin{lstlisting}[language=LuaRedis,firstnumber=30,
  caption={\srcnote{repository/cache/coupon.go:30--44 claimScript}}]
local stock = tonumber(redis.call('GET', KEYS[1]))
if stock == nil or stock <= 0 then
    return -1                                  -- 已抢光
end
local owned = tonumber(redis.call('GET', KEYS[2]))
if owned == nil then owned = 0 end
if owned >= tonumber(ARGV[1]) then
    return -2                                  -- 超出单人上限
end
redis.call('DECR', KEYS[1])
redis.call('INCR', KEYS[2])
redis.call('EXPIRE', KEYS[2], tonumber(ARGV[2]))
return 1
\end{lstlisting}
\end{frame}

\begin{frame}{Lua 脚本逐行解读}
\begin{itemize}
  \item \textbf{L1--L4}：读库存，\textbf{nil 当 0 处理}。返回 \texttt{-1} 给业务层翻译成 "已抢光"。
  \item \textbf{L5--L9}：读单人计数。第一次领该券时 key 不存在，Lua 里 \texttt{nil} 强转 0；命中上限直接返回 \texttt{-2}。
  \item \textbf{L10--L11}：DECR / INCR 在 Lua 里是同一个事务原子单位，不会出现"扣了库存但没记上单人"。
  \item \textbf{L12}：每次领到都续命 24h —— 防止 key 在活动跨天时被 evict 导致重新可领。
  \item 三个 return 码业务层映射成两种业务错误（见 \texttt{ClaimCouponAtomic}）。
\end{itemize}
\srcnote{repository/cache/coupon.go:57--75 错误码翻译}
\end{frame}

\section{抢红包：LPOP 一次取一份}
% ============================================================

\begin{frame}[fragile]{为什么红包不能用 DECR：发包时预拆 + 二倍均值法}
\textbf{业务约束}：总额严格守恒（100 元发出去必须正好 100 元），每份金额不同 $\to$ DECR 拿不回金额，必须 LPOP 出队拿"那一份"。\textbf{发包时一次性拆好}，写入 Redis LIST。

\begin{lstlisting}[language=Go,firstnumber=107,
  caption={\srcnote{repository/cache/redpacket.go:107--141 SplitRedPacket}}]
out := make([]int64, count)
remain := total
for i := 0; i < count-1; i++ {
    left := int64(count - i)
    avg := remain / left * 2          // 二倍均值
    if avg < 2 { avg = 2 }
    maxThisRound := remain - int64(count-1-i)  // 留至少 1 分给后面每份
    hi := avg - 1
    if hi > maxThisRound { hi = maxThisRound }
    if hi < 1 { hi = 1 }
    amt := splitRand.r.Int63n(hi) + 1  // [1, hi]
    out[i] = amt
    remain -= amt
}
out[count-1] = remain                  // 最后一份兜底
splitRand.r.Shuffle(count, ...)        // 打散顺序，避免位置敏感
\end{lstlisting}
\end{frame}

\begin{frame}{二倍均值逐行解读}
\begin{itemize}
  \item \textbf{avg = remain/left * 2}：剩余金额除以剩余份数 × 2，作为"本次最大可分到金额"的基础。
  \item \textbf{maxThisRound}：留给后面每份至少 1 分（避免最后几份被迫为 0）。
  \item \textbf{Int63n(hi)+1}：在 \texttt{[1, hi]} 闭区间随机，保证每份至少 1 分。
  \item \textbf{out[count-1] = remain}：最后一份直接吃剩，确保总和守恒。
  \item \textbf{Shuffle}：不打散的话，越靠前抢到的期望金额越大 —— 用户会推断"第一个抢的最大"。
  \item \textbf{splitRand 进程内独占 rand}：\texttt{math/rand} 全局源加锁，热点路径并发不友好。
\end{itemize}
\srcnote{repository/cache/redpacket.go:92--142}
\end{frame}

\begin{lstlisting}[language=LuaRedis,firstnumber=52,

    return -2                                 -- 已领过

    return -1                                 -- 已抢完

\begin{frame}{三段 Lua 的共同设计纪律}
\begin{enumerate}
  \item \textbf{失败也要原子}：判断条件不通过时\textbf{什么都不写}，直接 return 错误码（-1 / -2）。
  \item \textbf{回滚也是 Lua}：DB 失败时的补偿动作要走\textbf{同一种原子机制}，不能 Go 侧两次 Redis 调用。
  \item \textbf{TTL 永远要设}：尤其是抢/领标记类 key，否则用户停留态 key 会无限堆积。
  \item \textbf{返回值留语义}：用 \texttt{>0 / 0 / 负数} 区分成功金额、抢完、已抢过、超限，业务层 switch 即可。
  \item \textbf{进程内 rand}：发包拆分时\textbf{不要用 math/rand 全局}，热点路径并发争锁。
\end{enumerate}
\srcnote{repository/cache/redpacket.go:92--96 splitRand 局部源；coupon.go:30--44 错误码语义}
\end{frame}

\section{防黑产：分层与现状}

\begin{frame}{为什么营销是黑产首选 + 四层防线}
\textbf{黑产为何盯营销}：真金白银（券能转卖、红包能提现）+ 规则透明（必须对外通告）+ 多账号便宜（手机号能买）。本质是\textbf{把作弊单价拉到 > 收益}。

\vspace{1mm}

\toprule

\midrule

L2 网关   & IP TokenBucket         & 已接 & router.go:22 \\
L3 业务   & 用户 SlidingWindow     & 部分 & 秒杀 + 红包，coupon 缺 \\

\begin{frame}{面试 Q\&A（上）}
\textbf{Q1}：为什么 coupon 用 DECR，红包用 LPOP？\\
\hspace{4mm}\small A：券同质，整型计数器即可；红包每份金额不同，必须出队拿"那一份"。

\vspace{2mm}
\textbf{Q2}：coupon Lua 里"先查库存再查单人"能不能换顺序？\\
\hspace{4mm}\small A：可以换，但语义有别：先查单人会先消耗 user key 的读，库存爆满时反而更慢。当前顺序对"已抢光"这种最常见的失败短路更划算。

\vspace{2mm}
\textbf{Q3}：3/s/用户 这个数从哪来？\\
\hspace{4mm}\small A：真人点击 300--500ms 一次，3 留 50\% 余量。压测 46/45 误差 2.2\%。
\end{frame}

\begin{frame}{面试 Q\&A（下）}
\textbf{Q4}：红包 Lua 已经 LPOP 了，DB 落库失败怎么办？\\
\hspace{4mm}\small A：Saga 回滚 —— 跑 \texttt{rollbackRedPacketClaimScript}，\texttt{LPUSH} 金额回队头 + \texttt{HDEL} 撤销 claimed 标记，用户下次能再抢。

\vspace{2mm}
\textbf{Q5}：羊毛党用一万个账号刷 coupon，gomall 现在能挡住吗？\\
\hspace{4mm}\small A：挡不住。需要追加 \texttt{SlidingWindow\{coupon, 5/min, ByUser\}}。当前只能靠全局 TokenBucket 100 RPS/IP 拖速度。

\vspace{2mm}
\textbf{Q6}：秒杀为什么不挂 Idempotency？\\
\hspace{4mm}\small A：秒杀重复点击的语义就是"再试一次"，幂等命中反而误导用户。限流截掉就够了。
\end{frame}
