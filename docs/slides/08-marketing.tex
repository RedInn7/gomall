\section{优惠券：库存 + 限领 二维原子}
% ============================================================

\begin{frame}[fragile]{coupon Lua 脚本（业务画像 + 10 行实现）}
\small \texttt{coupon:batch:\{id\}:stock} 整型库存；\texttt{coupon:user:\{uid\}:batch:\{id\}} 单人计数 TTL 24h。Lua 必须一锤完成：库存 > 0 ？单人 < \texttt{PerUser} ？两道闸过了才 DECR + INCR。
\begin{lstlisting}[language=LuaRedis,firstnumber=30,
  caption={\srcnote{repository/cache/coupon.go:30--44 claimScript}}]
local stock = tonumber(redis.call('GET', KEYS[1]))
if stock == nil or stock <= 0 then
    return -1                                  -- 已抢光
end
local owned = tonumber(redis.call('GET', KEYS[2]))
if owned == nil then owned = 0 end
if owned >= tonumber(ARGV[1]) then
    return -2                                  -- 超出单人上限
end
redis.call('DECR', KEYS[1])
redis.call('INCR', KEYS[2])
redis.call('EXPIRE', KEYS[2], tonumber(ARGV[2]))
return 1
\end{lstlisting}
\end{frame}

\begin{frame}{Lua 脚本逐行解读}
\begin{itemize}
  \item \textbf{L1--L4}：读库存，\textbf{nil 当 0 处理}。返回 \texttt{-1} 给业务层翻译成 "已抢光"。
  \item \textbf{L5--L9}：读单人计数。第一次领该券时 key 不存在，Lua 里 \texttt{nil} 强转 0；命中上限直接返回 \texttt{-2}。
  \item \textbf{L10--L11}：DECR / INCR 在 Lua 里是同一个事务原子单位，不会出现"扣了库存但没记上单人"。
  \item \textbf{L12}：每次领到都续命 24h —— 防止 key 在活动跨天时被 evict 导致重新可领。
  \item 三个 return 码业务层映射成两种业务错误（见 \texttt{ClaimCouponAtomic}）。
\end{itemize}
\srcnote{repository/cache/coupon.go:57--75 错误码翻译}
\end{frame}

\section{抢红包：LPOP 一次取一份}
% ============================================================

\begin{frame}[fragile]{为什么红包不能用 DECR：发包时预拆 + 二倍均值法}
\textbf{业务约束}：总额严格守恒（100 元发出去必须正好 100 元），每份金额不同 $\to$ DECR 拿不回金额，必须 LPOP 出队拿"那一份"。\textbf{发包时一次性拆好}，写入 Redis LIST。

\begin{lstlisting}[language=Go,firstnumber=107,
  caption={\srcnote{repository/cache/redpacket.go:107--141 SplitRedPacket}}]
out := make([]int64, count)
remain := total
for i := 0; i < count-1; i++ {
    left := int64(count - i)
    avg := remain / left * 2          // 二倍均值
    if avg < 2 { avg = 2 }
    maxThisRound := remain - int64(count-1-i)  // 留至少 1 分给后面每份
    hi := avg - 1
    if hi > maxThisRound { hi = maxThisRound }
    if hi < 1 { hi = 1 }
    amt := splitRand.r.Int63n(hi) + 1  // [1, hi]
    out[i] = amt
    remain -= amt
}
out[count-1] = remain                  // 最后一份兜底
splitRand.r.Shuffle(count, ...)        // 打散顺序，避免位置敏感
\end{lstlisting}
\end{frame}

\begin{frame}{二倍均值逐行解读}
\begin{itemize}
  \item \textbf{avg = remain/left * 2}：剩余金额除以剩余份数 × 2，作为"本次最大可分到金额"的基础。
  \item \textbf{maxThisRound}：留给后面每份至少 1 分（避免最后几份被迫为 0）。
  \item \textbf{Int63n(hi)+1}：在 \texttt{[1, hi]} 闭区间随机，保证每份至少 1 分。
  \item \textbf{out[count-1] = remain}：最后一份直接吃剩，确保总和守恒。
  \item \textbf{Shuffle}：不打散的话，越靠前抢到的期望金额越大 —— 用户会推断"第一个抢的最大"。
  \item \textbf{splitRand 进程内独占 rand}：\texttt{math/rand} 全局源加锁，热点路径并发不友好。
\end{itemize}
\srcnote{repository/cache/redpacket.go:92--142}
\end{frame}

\begin{lstlisting}[language=LuaRedis,firstnumber=52,

    return -2                                 -- 已领过

    return -1                                 -- 已抢完

\begin{frame}{三段 Lua 的共同设计纪律}
\begin{enumerate}
  \item \textbf{失败也要原子}：判断条件不通过时\textbf{什么都不写}，直接 return 错误码（-1 / -2）。
  \item \textbf{回滚也是 Lua}：DB 失败时的补偿动作要走\textbf{同一种原子机制}，不能 Go 侧两次 Redis 调用。
  \item \textbf{TTL 永远要设}：尤其是抢/领标记类 key，否则用户停留态 key 会无限堆积。
  \item \textbf{返回值留语义}：用 \texttt{>0 / 0 / 负数} 区分成功金额、抢完、已抢过、超限，业务层 switch 即可。
  \item \textbf{进程内 rand}：发包拆分时\textbf{不要用 math/rand 全局}，热点路径并发争锁。
\end{enumerate}
\srcnote{repository/cache/redpacket.go:92--96 splitRand 局部源；coupon.go:30--44 错误码语义}
\end{frame}

\section{防黑产：分层与现状}

\begin{frame}{为什么营销是黑产首选 + 四层防线}
\textbf{黑产为何盯营销}：真金白银（券能转卖、红包能提现）+ 规则透明（必须对外通告）+ 多账号便宜（手机号能买）。本质是\textbf{把作弊单价拉到 > 收益}。

\vspace{1mm}

\toprule

\midrule

L2 网关   & IP TokenBucket         & 已接 & router.go:22 \\
L3 业务   & 用户 SlidingWindow     & 部分 & 秒杀 + 红包，coupon 缺 \\

\section{拼团 + 满减：营销玩法的两个常见扩展}
% ============================================================

\begin{frame}{拼团把"个体优惠"做成"社交裂变"}
\begin{itemize}
  \item 拼团的业务公式：\textbf{N 人成团 = 单价 × 0.6}（举例），凑不齐就退款；凑齐了所有人都享受团购价。
  \item 和优惠券、秒杀的最大差别：\textbf{结果不是抢的瞬间确定的}，而是\textbf{在时间窗结束后才知道}。
  \item 三个关键参数决定团是否能形成：\textbf{N}（成团人数 2/3/5）、\textbf{T}（时间窗 24h/48h）、\textbf{发起人特权}（能否提前结团 / 自动开二期）。
  \item 资金路径：开团瞬间\textbf{付全款}（不是定金），凑齐瞬间扣的就是团购价；流团了走批量退款。
  \item 状态机：\texttt{Open} → \texttt{Joining} → \texttt{Full} → \texttt{Success}；或 \texttt{Joining} → \texttt{Expired} → \texttt{Refunded}。和订单 \texttt{auto\_cancel\_at} 一个套路：\textbf{cron 兜底 + RMQ 延时双保险}。
  \item 并发要点：Redis \texttt{HSET groupon:\{id\}:members userId 1} + \texttt{HLEN} 判定人数。Lua 一锤 HEXISTS→HLEN→HSET，形状\textbf{和 coupon claim 几乎一样}，只是 INCR 换成 HSET。
\end{itemize}
\srcnote{路线图：service/groupon.go；参照 repository/cache/coupon.go:30--44 + service/order\_task.go cron}
\end{frame}

\begin{frame}[fragile]{满减 vs 折扣券：先算满减再算券}
\small 业务规则：\texttt{Type=1 满减} 按订单金额自动触发，\texttt{Type=2 折扣}用户领取后勾选使用。叠加时\textbf{先满减再券} —— 反过来用户感觉被坑（券打掉单价后凑不够档位）。多档满减按阈值倒序匹配，命中第一档即返回。
\begin{lstlisting}[language=Go,firstnumber=1,
  caption={阶梯满减按阈值倒序匹配 + 先满减再券}]
type FullReduceTier struct {
    Threshold int64 // 满 100 元
    Amount    int64 // 减 10 元
}

// Tiers 必须按 Threshold 降序排列（DB 查出时 ORDER BY threshold DESC）
func ApplyFullReduce(tiers []FullReduceTier, amount int64) int64 {
    for _, t := range tiers {
        if amount >= t.Threshold {
            return t.Amount
        }
    }
    return 0
}

func FinalPrice(tiers []FullReduceTier, couponOff, amount int64) int64 {
    after := amount - ApplyFullReduce(tiers, amount)
    if after < 0 { after = 0 }
    final := after - couponOff
    if final < 0 { final = 0 }
    return final
}
\end{lstlisting}
\srcnote{路线图：service/promotion.go；字段已在 model.CouponBatch.Type/Threshold/Amount}
\end{frame}

\begin{frame}{满减的"凑单"行为 + 三个业务陷阱}
\begin{itemize}
  \item 用户为凑满 200 减 30 硬塞一件 30 元小商品 —— \textbf{客单价提升的关键武器}。
  \item \textbf{陷阱 1}：\textbf{凑单退货导致主订单不满档}。业内主流：\textbf{部分退款时按比例摊销满减}（退款金额 = 原金额 × (1 - 满减率)），避免薅羊毛"退一件保留满减"。
  \item \textbf{陷阱 2}：\textbf{档位过密}（满 100 减 10、满 110 减 12、满 120 减 14）—— 用户决策疲劳反而不下单。建议\textbf{3 档以内 + 跨度明显}。
  \item \textbf{陷阱 3}：\textbf{多店铺归集口径}。是按订单总价还是单店分摊？店铺承担成本时一定是后者。
  \item \textbf{退款规则}：满减券一般不退、会员券一般退 —— 由\textbf{券类型}决定，对照 \texttt{model.UserCoupon.Status}。
\end{itemize}
\srcnote{对照 deck 09 订单状态机 Refunding / Refunded 节点：退款时需要重算满减摊销}
\end{frame}

  caption={ZINCRBY 累加销量；超过 N=1000 时裁掉尾部}]
-- KEYS[1] = leaderboard:day:20260516
-- ARGV[1] = productId   ARGV[2] = delta
-- ARGV[3] = maxKeep     ARGV[4] = ttlSec

    redis.call('ZREMRANGEBYRANK', KEYS[1], 0, n - tonumber(ARGV[3]) - 1)

redis.call('EXPIRE', KEYS[1], tonumber(ARGV[4]))

\begin{frame}{面试 Q\&A（二）：风控与限流}
\textbf{Q4}：3/s/用户 这个数从哪来？\\
\hspace{4mm}\small A：真人点击 300--500ms 一次，3 留 50\% 余量。压测 46/45 误差 2.2\%。\\
\hspace{4mm}\small \textbf{代码}：\texttt{routes/router.go:145--152}；压测 \texttt{stressTest/REPORT.md \S 5}。

\vspace{2mm}
\textbf{Q5}：羊毛党用一万个账号刷 coupon，gomall 现在能挡住吗？\\
\hspace{4mm}\small A：挡不住。需要追加 \texttt{SlidingWindow\{coupon, 5/min, ByUser\}}，再叠加风控四问决策树。当前只能靠全局 TokenBucket 100 RPS/IP 拖速度。\\
\hspace{4mm}\small \textbf{代码}：\texttt{routes/router.go:22, 70--73}。

\vspace{2mm}
\textbf{Q6}：秒杀为什么不挂 Idempotency？\\
\hspace{4mm}\small A：秒杀重复点击的语义就是"再试一次"，幂等命中反而误导用户。限流截掉就够了。\\
\hspace{4mm}\small \textbf{代码}：\texttt{routes/router.go:145--152}（无 Idempotency）。
\end{frame}

\begin{frame}{面试 Q\&A（三）：拼团与满减}
\textbf{Q7}：拼团时间窗到了部分用户没付怎么办？\\
\hspace{4mm}\small A：拼团强制开团瞬间付全款，不存在"没付"；T 到了凑不齐就批量退款。状态机 Joining → Expired → Refunded，由 cron + RMQ 延时双兜底。\\
\hspace{4mm}\small \textbf{代码}：路线图 \texttt{service/groupon.go}，参照 \texttt{service/order\_task.go} cron。

\vspace{2mm}
\textbf{Q8}：满减券和折扣券一起用，先算哪个？\\
\hspace{4mm}\small A：先算满减再算券。先扣大头，券作用在剩余金额上；反过来用户感觉被坑（券打掉单价后凑不齐档位）。\\
\hspace{4mm}\small \textbf{代码}：路线图 \texttt{service/promotion.go}，已有 \texttt{model.CouponBatch.Type/Threshold/Amount} 字段支撑。

\vspace{2mm}
\textbf{Q9}：满减 + 部分退款怎么处理？\\
\hspace{4mm}\small A：按比例摊销满减 —— 退款金额 = 原金额 × (1 - 满减率)，避免"退一件保留满减"的薅羊毛。\\
\hspace{4mm}\small \textbf{代码}：路线图，对照 deck 09 Refunding/Refunded 节点。
\end{frame}

\begin{frame}{面试 Q\&A（四）：排行榜 / 直播 / A/B}
\textbf{Q10}：排行榜怎么防止某些商品永远霸榜导致内存膨胀？\\
\hspace{4mm}\small A：\texttt{ZADD} 后顺手 \texttt{ZREMRANGEBYRANK} 裁掉最低分的，最多保留 Top 1000；TTL 控制冷榜过期。\\
\hspace{4mm}\small \textbf{代码}：路线图 \texttt{repository/cache/leaderboard.go}，Lua 模板与 \texttt{ratelimit.go:18--34} 一致。

\vspace{2mm}
\textbf{Q11}：直播打赏要不要做 Saga 回滚？\\
\hspace{4mm}\small A：不要。礼物特效一旦显示不可撤销，业务上等价于"用户接受了"。只需保证\textbf{先扣钱再发特效}的顺序，DB 失败时直接报错给用户，不回滚特效（特效本身还没发）。\\
\hspace{4mm}\small \textbf{代码}：业务路线图，骨架参照 \texttt{service/redpacket.go:42--103}。

\vspace{2mm}
\textbf{Q12}：A/B 测试的分桶为什么不能用 sessionId？\\
\hspace{4mm}\small A：同一用户多次访问要进同一桶，否则用户在不同访问看到的优惠不一致，转化率失真。userId 哈希才稳定。\\
\hspace{4mm}\small \textbf{代码}：业务路线图 \texttt{middleware/experiment.go}。
\end{frame}
