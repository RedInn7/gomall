\section{业务目的：传统支付吃不下的那批订单}
% ============================================================

\begin{frame}{为什么 gomall 引入 Web3 支付}
\begin{itemize}
  \item \textbf{加密原生用户}：DeFi 用户钱包里全是链上资产，刷卡通道用不上。
  \item \textbf{跨境结算}：传统通道 1--3\% 手续费 + T+1 到账，跨境争议解决依赖银行 SWIFT。
  \item \textbf{资金托管增信}：买卖双方互不信任的大额订单，把"中介"从银行换成审计过的合约。
  \item \textbf{业务收益}：开新渠道，不是替代旧通道，跑在传统 \texttt{/paydown} 旁边。
\end{itemize}
\srcnote{routes/router.go:113--127 paydown（fiat）与 paydown/crypto（web3）平行挂载}
\end{frame}

\begin{frame}[fragile]{Escrow.sol 关键路径：fundWithOrderID}
\begin{lstlisting}[language=Java,firstnumber=74,
  caption={\srcnote{contracts/Escrow.sol:74--82 fundWithOrderID}}]
function fundWithOrderID(bytes32 _orderID)
    external payable onlyBuyer inState(State.Created)
{
    if (msg.value != amount) revert WrongAmount(amount, msg.value);
    orderID = _orderID;
    state = State.Funded;
    emit Funded(msg.value);
    emit PaymentConfirmed(_orderID, msg.sender, msg.value);
}
\end{lstlisting}
\small \textbf{逐行解读}：
\begin{itemize}
  \item \texttt{onlyBuyer + inState(Created)}：只能 buyer 调，且只能在 Created 态调，状态机入口唯一。
  \item \texttt{msg.value != amount}：链上强校验金额，少 1 wei 也 revert，业务侧零信任。
  \item \texttt{orderID = \_orderID}：链下订单号写进 storage，listener 拿它 join 业务库。
  \item \texttt{emit PaymentConfirmed}：\textbf{indexed orderID} 让 listener 只订一个 topic 就能精确过滤。
\end{itemize}
\end{frame}

\section{钱包签名：EIP-191 personal\_sign + nonce 防重放}
% ============================================================

\begin{frame}{为什么不能把私钥交给后端}
\begin{itemize}
  \item 私钥 = 钱包的全部，给了后端就不再是"用户的钱包"。
  \item 业界做法：后端给一段消息 $\to$ 钱包客户端内签 $\to$ 后端 \texttt{ecrecover} 反推地址。
  \item 业务收益：
  \begin{itemize}
    \item \textbf{不可否认}：地址匹配 = 用户授权过；
    \item \textbf{零私钥泄漏面}：签名发生在钱包客户端内部；
    \item \textbf{防钓鱼}：消息明文人类可读，钱包弹窗能让用户看清自己在签什么。
  \end{itemize}
\end{itemize}
\srcnote{pkg/web3/signature/verify.go:1--13 包注释设计意图}
\end{frame}

\begin{frame}[fragile]{VerifyPersonalSign：EIP-191 + ecrecover 12 行}
\begin{lstlisting}[language=Go,firstnumber=31,
  caption={\srcnote{pkg/web3/signature/verify.go:31--67}}]
func VerifyPersonalSign(addr string, msg []byte, sig []byte) (bool, error) {
    if len(sig) != personalSigLen { return false, fmt.Errorf("...") }
    if !common.IsHexAddress(addr) { return false, errors.New("...") }
    rsv := make([]byte, personalSigLen)
    copy(rsv, sig)
    switch rsv[64] {
    case 27, 28: rsv[64] -= 27
    case 0, 1:   // already normalized
    default:     return false, fmt.Errorf("非法 v 值: %d", rsv[64])
    }
    hash := personalSignHash(msg)             // EIP-191 prefix + keccak256
    pubKey, err := crypto.SigToPub(hash, rsv) // ecrecover
    if err != nil { return false, err }
    recovered := crypto.PubkeyToAddress(*pubKey)
    expected  := common.HexToAddress(addr)
    return bytes.Equal(recovered.Bytes(), expected.Bytes()), nil
}
\end{lstlisting}
\small \textbf{逐行}：长度 65 (r+s+v) 必须严格；地址 hex 校验阻拦明显非法输入；拷贝 rsv 避免污染调用方 slice；v 归一兼容钱包/go-ethereum 两种语义；\texttt{personalSignHash} 内部 \texttt{"\textbackslash x19Ethereum Signed Message:\textbackslash n"+len+msg} 前缀 + keccak256；\texttt{SigToPub} = ecrecover；最后 byte-level 比对地址（\texttt{HexToAddress} 自带 EIP-55 checksum 规整，大小写无关）。
\end{frame}

\begin{frame}[fragile]{StartPaymentListener 主循环 + connectAndServe}
\begin{lstlisting}[language=Go,firstnumber=177,
  caption={\srcnote{service/web3/listener.go:177--230 connectAndServe}}]

defer client.Close()
head, _ := client.BlockNumber(ctx)
last := l.loadLastBlock(ctx)
if last == 0 || last > head { last = head }
if last < head {
    l.catchUp(ctx, client, last+1, head)   // 回放停机期间漏掉的事件

l.saveLastBlock(ctx, head)
logsCh := make(chan types.Log, subscribeBuffer)   // = 64

    l.query(big.NewInt(int64(head+1)), nil), logsCh)
defer sub.Unsubscribe()
l.backoff.reset()                                // 连上立即清退避
for {
    select {
    case <-ctx.Done():    return ctx.Err()
    case err := <-sub.Err(): return err           // -> 外层重连
    case lg := <-logsCh:
        if lg.Removed { continue }                // reorg 撤销，跳过

        l.saveLastBlock(ctx, lg.BlockNumber)

\section{Gas 与费用模型：每笔订单实际花多少链费}
% ============================================================

\begin{frame}{Gas 概念：合约执行的"计价器"}
\begin{itemize}
  \item \textbf{gas} 是 EVM 操作的计量单位（SLOAD 2100、SSTORE 20000、转账约 21000）。
  \item 一笔 tx 的链费 $= \text{gasUsed} \times \text{gasPrice}$，以 wei 计；mainnet 普通转账 21000 gas，按 30 gwei 算 $\approx 0.00063$ ETH。
  \item EIP-1559 起把 \texttt{gasPrice} 拆成 \textbf{baseFee}（协议销毁，按区块占用率自适应）+ \textbf{priorityFee}（小费给打包者）。
  \item gomall 链费走\textbf{用户自付}：钱包弹窗里用户清楚看到 gas estimate，避免后端补贴被刷。
\end{itemize}
\srcnote{contracts/Escrow.sol:32--35 custom error 替代 require 字符串（gas 友好）}
\srcnote{contracts/README.md:91--92 release/refund 用 call 不用 transfer}
\end{frame}

\begin{frame}[fragile]{对比：require 字符串 vs custom error 的 gas 差异}
\begin{lstlisting}[language=Java,firstnumber=1,
  caption={\srcnote{contracts/Escrow.sol:32--35 custom error；:69 revert WrongAmount}}]
// 旧风格（gas 高）：失败时 ABI-encode 一整段字符串
function fund() external payable {
    require(msg.value == amount, "wrong amount: must equal escrow value");
    // ...
}

// gomall 当前风格（gas 友好）：4 字节 error selector + 紧凑参数
error WrongAmount(uint256 expected, uint256 actual);
function fundWithOrderID(bytes32 _orderID)
    external payable onlyBuyer inState(State.Created)
{
    if (msg.value != amount) revert WrongAmount(amount, msg.value);
    // ... 后续状态推进
}
\end{lstlisting}

\end{frame}

\section{多签仲裁：从单 EOA 升到 2-of-3 多签}
% ============================================================

\begin{frame}{单签 arbiter 的业务风险}
\begin{itemize}
  \item 当前 MVP：\texttt{arbiter} 是平台一个 EOA，私钥放运营手里。
  \item \textbf{业务风险}：私钥丢失 = 所有 Disputed 订单卡死；私钥被盗 = 攻击者可以无理由把所有托管金 \texttt{refund} 到自己控制的地址（因为 buyer 地址是攻击者）— 等等，实际上 refund 只会进 buyer，但仲裁人可以\textbf{偏向裁决}，相当于平台跑路风险。
  \item \textbf{对外承诺}：\$10{,}000+ 的订单用户不会接受 "平台单签拍板"；监管也不接受。
  \item 升级：把 arbiter 换成 \textbf{Gnosis Safe 2-of-3 多签}，3 个签名人持有，任意 2 个签名才能 release / refund。
\end{itemize}
\srcnote{contracts/Escrow.sol:10 arbiter immutable；:85--100 release / refund 入口}
\srcnote{contracts/README.md:101--102 arbiter 推荐用多签}
\end{frame}

\begin{frame}[fragile]{Safe 调 Escrow.release 的 calldata 模板}
\begin{lstlisting}[language=Java,firstnumber=1,
  caption={\srcnote{contracts/Escrow.sol:85--91 release()；通过 Safe 转发}}]
// Step 1: 在 Safe 后台或 SDK 里提案一笔交易
//   to:    address(escrow)                    // Escrow 合约地址
//   value: 0                                  // release 不带 ETH
//   data:  abi.encodeWithSignature("release()") // 4 字节 selector
//   operation: Call (0)                       // 普通 call，非 DelegateCall

// Step 2: 收集 2 个签名（owner 私钥 EIP-712 签 SafeTx hash）
// Step 3: 任一签名人调 Safe.execTransaction(...)
//   Safe 内部做：
//     1) 校验 threshold 个签名 (2 of 3)
//     2) 检查 nonce 单调递增 (防重放)
//     3) call escrow.release()，此时 msg.sender = Safe
// Step 4: Escrow.release() 校验 msg.sender == arbiter (= Safe addr) 通过
\end{lstlisting}
\small \textbf{逐行}：Safe 本质是个\textbf{通用 multisig 合约钱包}，对外暴露 \texttt{execTransaction}，要求 N-of-M 个 owner EIP-712 签名后才转发 calldata；从 Escrow 视角看 \texttt{msg.sender} 就是 Safe 地址（一个合约地址），跟 EOA 没区别；
nonce 单调递增防的是\textbf{同一组签名换块重放}（提案 N 用完后 nonce++，旧签名失效）；阈值变更走 Safe 自己的治理（\texttt{changeThreshold} 也要 N-of-M 签）；
gomall 落地：MVP 阶段 arbiter 还是 EOA，灰度后切到 Safe \textbf{无需重新部署 Escrow}，只在新订单的 constructor 里换 arbiter 地址即可。
\end{frame}

必须：(1) 先调 USDC 合约的 \texttt{approve(escrow, N)} 授权 — 在 USDC 内部记 \texttt{allowance[buyer][escrow] = N}；(2) 再调 Escrow 的 \texttt{fundWithOrderID}，里面用 \texttt{IERC20.transferFrom(buyer, this, N)} 拉款。两步 tx，用户得签两次。

\begin{frame}[fragile]{EscrowERC20.fundWithOrderID 关键路径}
\begin{lstlisting}[language=Java,firstnumber=1,
  caption={\srcnote{contracts/EscrowERC20.sol（规划）— 基于 Escrow.sol:76--82 改造}}]
import {IERC20} from "@openzeppelin/contracts/token/ERC20/IERC20.sol";

contract EscrowERC20 {
    IERC20 public immutable token;       // USDC / USDT / DAI 合约地址
    uint256 public immutable amount;     // USDC 的 smallest unit (6 位小数)
    // buyer / seller / arbiter / state / orderID 同 Escrow.sol

    function fundWithOrderID(bytes32 _orderID)
        external onlyBuyer inState(State.Created)
    {
        orderID = _orderID;
        state = State.Funded;
        // 关键：transferFrom 必须先 approve 过；返回 false 也算失败
        bool ok = token.transferFrom(msg.sender, address(this), amount);
        require(ok, "transferFrom failed");
        emit Funded(amount);
        emit PaymentConfirmed(_orderID, msg.sender, amount);
    }
}
\end{lstlisting}
\small \textbf{逐行}：去掉 \texttt{payable}，因为不再收原生币；amount 单位变成 USDC 的 6 位小数（\$1 = 1{,}000{,}000）；\texttt{transferFrom} 的 ok 必须显式 check — USDT 早期版本不返回 bool，要用 OpenZeppelin SafeERC20；
\textbf{先改 state 再 transferFrom} 是有意为之（CEI 模式：Checks-Effects-Interactions），万一 token 是恶意合约重入回来，state 已经是 Funded，撞 \texttt{inState(Created)} 直接 revert；
event 完全复用，listener 不用改一行代码就支持 USDC 路径。
\end{frame}

\begin{frame}{reorg 是什么 + 链是有"概率最终性"的}
\begin{itemize}
  \item PoS 以太坊：每 12s 出一个块；\textbf{每个 epoch（32 块 $\approx$ 6.4 分钟）finalize 一次}，finalize 后理论上不再回滚（除非 1/3 验证人作恶 = 损失数十亿美元的 stake）。
  \item finalize 之前的块都是"软共识"，遇到分叉时短链会被丢弃 = reorg。
  \item 业务影响：如果在 1 块确认时就 \texttt{release}，下一秒被 reorg 撤销 = 卖家收到的钱"凭空消失"。
  \item 对策：按\textbf{订单金额}动态选 N 确认。
\end{itemize}

\begin{frame}{面试 Q\&A（上）}
\textbf{Q1}：链 reorg 把已 release 的资金撤销了怎么办？\\
\hspace{4mm}\small A：金额越大 N 确认越多（mainnet 大额走 32 块 = 1 epoch finalize；L2 大额等回 L1 finalize）；listener 监听 \texttt{lg.Removed} 跳过 reorg log，订单状态不会被错误推进；已发货走对账兜底 + 仲裁人介入。\\
\srcnote{service/web3/listener.go:218--221}

\vspace{1.5mm}
\textbf{Q2}：用户钱包私钥丢了订单怎么处理？\\
\hspace{4mm}\small A：链上没有"重置密码"。资金在 Funded 态可由仲裁人核实身份后 refund 到新地址；已 Released 给商家则只能走链下补偿。\\
\srcnote{contracts/Escrow.sol:94--100 refund 入口可由 arbiter 触发}

\vspace{1.5mm}
\textbf{Q3}：gas 飙升导致用户支付 tx 失败怎么补救？\\
\hspace{4mm}\small A：默认路由到 L2；nonce 5min 内允许用户重新提交带更高 gas 的 tx；超时则重新颁发 nonce + 签名。EIP-1559 钱包会自动调高 maxFee 减少这种失败。\\
\srcnote{repository/cache/web3.go:21--26 Web3NonceTTL=5min}

\vspace{1.5mm}
\textbf{Q4}：custom error 和 require 字符串对 gas 影响有多大？\\
\hspace{4mm}\small A：单次 revert 节省 $\approx$ 50 gas；mainnet 30 gwei 下相当于每笔失败 tx 省 0.0000015 ETH。看似小，但合约一年 100 万次 revert 就是 1.5 ETH，且 calldata 更短利于前端解码。\\
\srcnote{contracts/Escrow.sol:32--35 错误定义；:69 revert WrongAmount}
\end{frame}

\begin{frame}{面试 Q\&A（中）}
\textbf{Q5}：和传统第三方支付的对账如何并存？\\
\hspace{4mm}\small A：两条通道都把支付事件落到同一条 outbox 总线，事件加 channel 字段（fiat / web3-eth / web3-usdc）。对账任务按 \texttt{order\_id} join 法币入账 + 链上 confirmed event。\\
\srcnote{service/payment\_crypto.go:129--144 outbox 写入逻辑}

\vspace{1.5mm}
\textbf{Q6}：为什么 nonce 必须后端发，前端自己生成不行吗？\\
\hspace{4mm}\small A：前端可生成的 nonce 后端没法保证唯一性也没法做时效控制；攻击者拿到一次有效签名后可无限重放。后端 Redis + Lua 原子 GET+DEL 才能保证"一次性"。\\
\srcnote{repository/cache/web3.go:60--89 consumeNonceScript}

\vspace{1.5mm}
\textbf{Q7}：listener 进程重启会不会丢事件？\\
\hspace{4mm}\small A：不会。\texttt{loadLastBlock} 从 Redis 拿到最后处理过的块号，\texttt{catchUp(last+1, head)} 用 \texttt{FilterLogs} 批量回放停机期漏掉的事件，再叠加 SETNX 72h 幂等，重启不会重做也不会丢做。\\
\srcnote{service/web3/listener.go:188--197 catchUp；:316--326 tryClaim}

\vspace{1.5mm}
\textbf{Q8}：单签 arbiter 上线后多久应该切多签？\\
\hspace{4mm}\small A：单笔金额 \$100 以内可继续单 EOA；超过 \$100 或托管总额超过 \$10k 强烈建议切 2-of-3 Gnosis Safe；切换不需重新部署 Escrow，只在新订单的 constructor 里传 Safe 地址即可。\\
\srcnote{contracts/Escrow.sol:57--65 constructor；contracts/README.md:101--102}
\end{frame}

\begin{frame}{面试 Q\&A（下）}
\textbf{Q9}：稳定币集成的最大业务卡点是什么？\\
\hspace{4mm}\small A：两步 tx (approve + transferFrom) 用户体验差。落地路径：(1) 短期接受两步 tx，前端引导用户一次授权较大额度；(2) 中期接 EIP-2612 \texttt{permit}，把 approve 改成一次签名同 tx 内做 transferFrom；(3) 长期接 ERC-4337 账户抽象，把多笔操作合成一笔。\\
\srcnote{contracts/EscrowERC20.sol（规划）— 基于 Escrow.sol:76--82 改造}

\vspace{1.5mm}
\textbf{Q10}：Optimistic rollup 的 7 天挑战期对 gomall 有什么实际影响？\\
\hspace{4mm}\small A：对 L2 内闭环交易\textbf{无影响}（买家 L2 入、卖家 L2 出，资金一直在 L2 上）；只有当卖家想把 L2 的稳定币提回 L1 时才碰到 7 天。变通：走第三方 fast-bridge（Hop / Across），用流动性提供方贴前 7 天的钱给卖家，平台 / LP 收 0.1\%--0.3\% 手续费。\\
\srcnote{contracts/README.md:96--100 escrow 在本链闭环}

\vspace{1.5mm}
\textbf{Q11}：RPC 节点中途切换会不会导致漏事件？\\
\hspace{4mm}\small A：不会。\texttt{last\_block} 持久化在 Redis，换源后 \texttt{loadLastBlock} + \texttt{catchUp} 自动补齐换源期间的所有 log；再叠加 SETNX 72h 幂等，重复推送也只生效一次。\\
\srcnote{service/web3/listener.go:328--352 loadLastBlock / saveLastBlock}

\vspace{1.5mm}
\textbf{Q12}：合规 (OFAC) 名单上的钱包地址来 paydown 怎么拦？\\
\hspace{4mm}\small A：在 \texttt{VerifyAndPark} 第 2 段（验签后、写 outbox 前）插一个 Chainalysis / TRM Labs 黑名单查询，命中则返回 \texttt{W3-COMPLIANCE-REJECTED} + 客服话术；不修改合约，纯链下卡口，合规可独立审计。\\
\srcnote{service/payment\_crypto.go:116--124 验签后是天然过滤点}
\end{frame}
