\section{为什么 3 态不够用}
% ============================================================

\begin{frame}{deck 15 已经把"关单"讲透，但只覆盖了 1 个分支}
\begin{itemize}
  \item gomall 直到这一版之前只用了 3 个状态：\textbf{UnPaid / Paid / Cancelled}。
  \item deck 15 详细讲了 30min 关单链路 —— 这是\texttt{UnPaid → Cancelled} 这一条边。
  \item 真实电商订单的生命周期至少有 7 态：付款、发货、收货、退款各自是单独的合同节点。
  \item 缺位带来的业务后果：
  \begin{itemize}
    \item \textbf{付款后无法跟踪履约}：商家发货 / 用户收货没有专门的状态。
    \item \textbf{售后无法承接}：退款只能借用旧字段或脱离状态机硬改。
    \item \textbf{数据指标失真}：成交转化、退款率、自动好评率全都没有口径。
  \end{itemize}
\end{itemize}
\srcnote{deck 15 order-close-business 是本 deck 的前置；本 deck 在 v2.1 把状态机补齐到 7 态}
\end{frame}

func CanTransition(from, to uint) bool {
    allowed, ok := orderStateTransitions[from]
    if !ok {
        return false
    }
    for _, s := range allowed {

  \item \textbf{已付款} ≠ \textbf{已发货}：业务上要把两件事拆开，否则商家压货 / 用户催单都没数据抓手。
  \item 发货事件 \texttt{order.shipped} 会触发：物流系统对接、推送"已发货"、平台 7d 倒计时启动。
  \item gomall 当前不持久化物流单号（model 没字段），\textbf{tracking 仅经事件透传}；下游服务自行落表。
\end{itemize}

\begin{frame}[fragile]{ShipOrder 实现：把规则、SQL、事件原子绑死}
\begin{lstlisting}[language=Go,firstnumber=36,
  caption={service/order\_shipping.go:36--71 ShipOrder 主体},captionpos=b]
func (s *ShippingSrv) ShipOrder(ctx context.Context, orderNum uint64, trackingNo, carrier string) error {
    baseDao := dao.NewOrderDao(ctx)
    order, err := baseDao.GetOrderByOrderNum(orderNum)
    if err != nil { return err }
    if order == nil || order.ID == 0 { return errors.New("订单不存在") }
    if order.Type != consts.OrderWaitShip {
        return ErrInvalidOrderStateTransition
    }
    return baseDao.DB.Transaction(func(tx *gorm.DB) error {
        ok, err := dao.NewOrderDaoByDB(tx).ShipOrder(orderNum)
        if err != nil { return err }
        if !ok { return ErrInvalidOrderStateTransition }
        return dao.NewOutboxDaoByDB(tx).Insert(
            "order", "OrderShipped", "order.shipped", order.ID,
            events.OrderShipped{...})
    })
}
\end{lstlisting}
\begin{itemize}
  \item \textbf{先读后写不可信}：所以 DAO 里再用 WHERE 死锁 from=WaitShip，避免并发覆盖。
  \item DB tx 包住"状态变更 + outbox 写入"——outbox 模式保证最终一致。
  \item \textbf{tracking / carrier 进事件不进表}：本期 model 不动，路线图标注引入 \texttt{shipping\_info} 子表。
\end{itemize}
\srcnote{service/order\_shipping.go:36--71；repository/db/dao/order.go:202--210 条件 UPDATE}
\end{frame}

\begin{frame}[fragile]{DAO 层条件 UPDATE：状态机的最后一道闸}
\begin{lstlisting}[language=Go,firstnumber=202,
  caption={repository/db/dao/order.go:202--210 ShipOrder DAO},captionpos=b]
func (dao *OrderDao) ShipOrder(orderNum uint64) (bool, error) {
    res := dao.DB.Model(&model.Order{}).
        Where("order_num=? AND type=?", orderNum, consts.OrderWaitShip).
        Update("type", consts.OrderWaitReceive)
    if res.Error != nil { return false, res.Error }
    return res.RowsAffected > 0, nil
}
\end{lstlisting}
\begin{itemize}
  \item \texttt{WHERE type=WaitShip} 兜住所有并发场景：第二次重发只会 \texttt{RowsAffected=0}。
  \item 这条 SQL = 状态机推进的\textbf{原子保证}，service 层就算被绕过也不会破坏数据一致性。
  \item 模式和 deck 15 讲的 \texttt{CloseOrderWithCheck} 一致 —— gomall 在每条边都重复这个模式。
\end{itemize}
\srcnote{repository/db/dao/order.go:202--210 ShipOrder；:214--222 ConfirmReceive；:240--248 ApproveRefund；:185--196 CloseOrderWithCheck 对照}
\end{frame}

\begin{frame}[fragile]{RunAutoConfirmReceive：cron 兜底的标准模式}

func (s *OrderTaskService) RunAutoConfirmReceive() {
    ctx := context.Background()
    baseDao := dao.NewOrderDao(ctx)
    orders, err := baseDao.GetTimeoutWaitReceive(autoConfirmReceiveDays, 100)
    if err != nil {
        util.LogrusObj.Errorf("Cron Job Error: fetch wait-receive orders failed: %v", err)
        return

    }

\section{Refund 三态：业务对资金 / 评价 / 物流的强约束}
% ============================================================

\begin{frame}{Refunding / Refunded 为什么分两个态}
\begin{itemize}
  \item \texttt{Refunding}：用户已发起退款，但运营 / 商家\textbf{还没批}。
  \begin{itemize}
    \item 评价系统：冻结评分，避免"用户退款 → 还给差评"的尴尬。
    \item 物流系统：如果还在 WaitShip 期间发起，商家不能再发货。
    \item 客服系统：自动派单等待处理。
  \end{itemize}
  \item \texttt{Refunded}：钱真退到用户账上 → 才能触发资金扣回 / 库存对账。
  \item 业务上\textbf{发起退款} ≠ \textbf{退款成功}：拆开是为了让\textbf{每一步都有专门的下游消费者}。
\end{itemize}
\keypoint{把"申请"和"完成"拆开，是大多数金融型状态机的通用模式 (订单 / 提现 / 红包都一样)。}
\srcnote{service/refund.go:46--91 RequestRefund + 93--128 ApproveRefund 是两条独立事件链}
\end{frame}

\begin{frame}[fragile]{RequestRefund：from 集合是业务规则，不能写在 service 入口}
\begin{lstlisting}[language=Go,firstnumber=36,
  caption={service/refund.go:36--40 + DAO},captionpos=b]
var refundAllowedFrom = []uint{
    consts.OrderWaitShip,
    consts.OrderWaitReceive,
    consts.OrderCompleted,
}
// DAO 用 WHERE type IN (?) 一次拦截非法 from
func (dao *OrderDao) RequestRefund(orderNum uint64, allowedFrom []uint) (bool, error) {
    res := dao.DB.Model(&model.Order{}).
        Where("order_num=? AND type IN ?", orderNum, allowedFrom).
        Update("type", consts.OrderRefunding)
    return res.RowsAffected > 0, res.Error
}
\end{lstlisting}
\begin{itemize}
  \item 退款入口有\textbf{多个 from}（不同业务场景）——\texttt{WHERE type IN (?)} 把"读后写竞态"消灭在 SQL 层。
  \item 未付款 (\texttt{WaitPay}) 不在列表里：应走 \texttt{CancelUnpaidOrder} 关单链路，不是退款。
  \item \texttt{Closed / Refunded} 不在列表里：终态不可逆。
\end{itemize}
\srcnote{service/refund.go:36--40 集合；service/refund.go:46--91 RequestRefund 主流程；repository/db/dao/order.go:226--238 DAO}
\end{frame}

\begin{frame}{ApproveRefund 本期只走状态机，不真退钱}
\begin{itemize}
  \item gomall v2.1 的 \texttt{ApproveRefund}\textbf{只推状态机} + 写 \texttt{order.refunded} 事件。
  \item 真退钱在下游：
  \begin{itemize}
    \item \textbf{法币}：钱包服务消费事件 → 调用支付通道退款接口（占位 \texttt{tx\_id}）。
    \item \textbf{Web3}：listener 消费事件 → 链上转账后回写 \texttt{tx\_id}。
  \end{itemize}
  \item 这是\textbf{编排型 saga}：每一步都通过事件触发，避免在主链路里同步阻塞第三方。
  \item 路线图：等钱包服务落地后，把 \texttt{TxID} 字段填上，关闭"approve 后钱还没退"的 gap。
\end{itemize}
\srcnote{service/refund.go:93--128 ApproveRefund；events/events.go OrderRefundedEvent.TxID 占位}
\end{frame}

\begin{frame}{RejectRefund：唯一允许"回到 Completed"的边}
\begin{tikzpicture}[node distance=6mm and 14mm]
  \node[node-state-alert] (rg) {Refunding};
  \node[node-state-normal,right=of rg] (cp) {Completed};
  \node[node-state-transition,below=of rg] (rd) {Refunded};
  \draw[arrow,dashed] (rg) -- node[above,font=\tiny]{Reject (回 Completed)} (cp);
  \draw[arrow] (rg) -- node[left,font=\tiny]{Approve} (rd);
\end{tikzpicture}
\vspace{2mm}
\begin{itemize}
  \item Refunding → Completed 这条边\textbf{看上去像状态回退}，但实际上是业务上完全合法的"售后未通过"。
  \item 业务后果：用户能再次发起退款（重新走一遍流程）；评价系统解冻；客服工单关闭。
  \item DAO 层依然用 \texttt{WHERE type=Refunding} 兜底，避免误把 WaitShip 或 WaitReceive 直接推到 Completed。
\end{itemize}

\end{frame}

\section{物流单独立表设计}
% ============================================================

\begin{frame}{物流不入 order 主表的业务依据}
\begin{itemize}
  \item gomall 当前 \texttt{model.Order} 没有 \texttt{tracking\_no / carrier} 字段：发货事件只把单号透传出去。
  \item 业务上一个订单可能对应\textbf{多个物流包裹}：跨仓拆单 / 大件分箱 / 预售分批，必须 1:N。
  \item 物流商有自己的状态机（揽收 / 转运 / 派送 / 签收 / 拒收），节奏远高于 order 主表更新频率。
  \item 主表保持精简，物流相关字段下沉到 \texttt{logistics\_order} 子表，主表只持 \texttt{order\_num} 关联。
\end{itemize}
\keypoint{把高频物流节点和低频订单状态切开，是大流量电商写多读多场景的标准切分。}
\srcnote{repository/db/model/order.go:7--18 当前 model；service/order\_shipping.go:36 ShipOrder 把 tracking 直接放进事件}
\end{frame}

  \item \textbf{PreOrder (预售)}：\texttt{WaitPay} 之前加 \texttt{Preordering}，订金已付 / 尾款未付。
  \item \textbf{Reservation (预约)}：服务类订单专用，\texttt{WaitShip} 替换为 \texttt{Scheduling}（约时间），\texttt{WaitReceive} 替换为 \texttt{Serving}（服务中）。
  \item \textbf{Subscription (订阅)}：父订单恒定 \texttt{Active}，每期子订单走完整 7 态；本质是聚合根 + 周期。
\end{itemize}
\keypoint{扩展原则：\textbf{加节点不改老节点 / 加子状态机不动主表 / 加 const 不改数值}。}
\srcnote{路线图：consts/order.go 数值留出 10-19 给预售，20-29 给预约，30-39 给订阅}
\end{frame}

% 状态机变更的兼容策略 (合并入兼容性章节)

\begin{frame}{为什么"DB 数值不动 + 别名"是兼容的最低成本}
\begin{itemize}
  \item DB 里有\textbf{数百万存量 order}：\texttt{type} 字段是 tinyint，改值需要 \texttt{ALTER + UPDATE} 全表扫，停机不可接受。
  \item 老旧客户端（小程序 / Android / 历史 API）还在用旧字段名 \texttt{UnPaid / Paid / Cancelled} 拼前端字典。
  \item 兼容策略：
  \begin{itemize}
    \item DB 数值\textbf{完全不动}。
    \item 旧名变 \texttt{Deprecated} 别名，新代码用新名。
    \item 旧前端 \texttt{OrderTypeMap} 并存，转译同一组 int。
  \end{itemize}
\end{itemize}
\keypoint{常量层的别名 + 编译期校验 = 零停机 / 零回归的迁移。}
\srcnote{consts/order.go:20--38 别名块；consts/order.go:51--57 旧 OrderTypeMap 保留}
\end{frame}

% Cron 自动确认深挖 (合并入 ConfirmReceive 章节)

\begin{frame}{商家可配 + 节假日延期 + 灾备}
\begin{itemize}
  \item 商家\textbf{可配 SLA}：服装类 7d / 大件家电 15d / 跨境 30d；落地后改成 \texttt{merchant.auto\_confirm\_days} 查表。
  \item \textbf{节假日延期}：春节 / 国庆物流积压期把 7d 自动延长到 10d，避免大批量误判。
  \item \textbf{灾备}：cron 节点单点宕掉怎么办？
  \begin{itemize}
    \item 跑\textbf{多副本}：用 redis SETNX 抢锁，只让一个实例真跑。
    \item 跑\textbf{失败补偿}：下次拉的是 \texttt{updated\_at} 落后窗口的所有订单，天然兜底。
    \item 跑\textbf{延后补偿}：如果 cron 持续宕，订单仍停在 \texttt{WaitReceive}，不会推错。
  \end{itemize}
\end{itemize}
\keypoint{cron 兜底的安全性不靠"必须按时跑"，靠\textbf{推迟也无害 + 多跑也幂等}的设计。}
\srcnote{initialize/cron.go:42--52 注册 6h 一跑；service/order\_task.go:69--85 单笔事务 + 幂等}
\end{frame}

% 退款链路分析 (合并入 Refund 三态章节)

\begin{frame}{原路退回 vs 钱包余额退}
\begin{itemize}
  \item \textbf{原路退回}（默认）：按支付方式逆向：支付宝退支付宝 / 微信退微信 / Web3 退原链。
  \begin{itemize}
    \item 优点：合规 / 用户最少惊讶 / 不用银行进行二次清算。
    \item 缺点：支付通道有自己的退款 SLA（1--15 天），出钱慢。
  \end{itemize}
  \item \textbf{钱包余额退}：把钱退到平台站内余额。
  \begin{itemize}
    \item 优点：秒到 / 鼓励二次消费 / 平台沉淀资金。
    \item 缺点：合规要看地区 / 用户可能要求强制走原路。
  \end{itemize}
  \item gomall 当前 \texttt{OrderRefundedEvent.TxID} 留空：下游钱包 / 支付服务消费事件后决定走哪条路。
\end{itemize}
\srcnote{events/events.go OrderRefundedEvent.TxID 占位；钱包 / 支付服务消费决策}
\end{frame}

\begin{frame}{业务 Q\&A（上）}
\textbf{Q1}：为什么旧代码不直接重命名 \texttt{UnPaid → OrderWaitPay}？\\
\hspace{4mm}\small A：怕一次改面太大引入 bug。先用 \texttt{const} 别名跑通编译 + 单测，老引用慢慢迁。\\
\hspace{4mm}\srcnote{consts/order.go:20--38}

\vspace{1mm}
\textbf{Q2}：物流单号不入表，下游怎么消费？\\
\hspace{4mm}\small A：\texttt{order.shipped} 事件里带 \texttt{tracking\_no / carrier}，下游物流服务自行落表；outbox 模式标准操作。\\
\hspace{4mm}\srcnote{service/order\_shipping.go:36--71；events/events.go OrderShipped}

\vspace{1mm}
\textbf{Q3}：7d 自动确认收货被用户投诉了怎么办？\\
\hspace{4mm}\small A：状态机允许 \texttt{Completed → Refunding}，售后窗口期内仍可发起退款。Completed 不是死刑。\\
\hspace{4mm}\srcnote{service/order\_state.go:24}

\vspace{1mm}
\textbf{Q4}：商家发货后用户不收货，能让商家主动确认吗？\\
\hspace{4mm}\small A：不允许，会被薅羊毛（自买自卖确认后立即结算）。必须走 7d cron 兜底。\\
\hspace{4mm}\srcnote{service/order\_task.go:60--90 cron 是唯一入口}

\vspace{1mm}
\textbf{Q5}：Refunding 状态商家先于运营批准发货了怎么办？\\
\hspace{4mm}\small A：\texttt{ShipOrder} DAO 层 \texttt{WHERE type=WaitShip}，Refunding 不匹配 \texttt{RowsAffected=0}。商家发货被拒。\\
\hspace{4mm}\srcnote{repository/db/dao/order.go:202--210}
\end{frame}

\begin{frame}{业务 Q\&A（中）}
\textbf{Q6}：为什么 \texttt{ApproveRefund} 不真退钱？\\
\hspace{4mm}\small A：主链路不阻塞第三方支付。状态推进 + outbox 后，钱包 / 支付通道服务异步消费事件真退钱并回填 \texttt{tx\_id}。saga 风格。\\
\hspace{4mm}\srcnote{service/refund.go:93--128}

\vspace{1mm}
\textbf{Q7}：cron 每 6 小时跑一次会不会太稀疏？\\
\hspace{4mm}\small A：自动确认收货 = \textbf{资金结算触发器}，不要求秒级。6 小时一次足够；过于频繁反而徒增 DB 压力。\\
\hspace{4mm}\srcnote{initialize/cron.go:42--52}

\vspace{1mm}
\textbf{Q8}：部分退款 / 部分发货什么时候支持？\\
\hspace{4mm}\small A：等多 SKU 订单聚合根落地。届时 line item 表自己一套小状态机，聚合根 7 态保留不变。\\
\hspace{4mm}\srcnote{service/refund.go:105 amount 取全单总额，路线图项}

\vspace{1mm}
\textbf{Q9}：3 态 → 7 态的迁移要不要发停机公告？\\
\hspace{4mm}\small A：不用。存量 DB 数据完全兼容，常量层只是 alias；编译期就能验证全代码没改语义。\\
\hspace{4mm}\srcnote{consts/order.go:6--38}
\end{frame}
