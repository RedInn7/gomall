\section{第一层：TokenBucket 全局兜底}
% ============================================================

\begin{frame}{为什么全局还要 100 RPS / 200 burst}
\begin{itemize}
  \item gomall 单机基线 \textbf{64{,}254 RPS}（\texttt{/ping}），单 IP 真正用户用不到 1/600。
  \item 100 RPS / 200 burst 实际拦的是\textbf{爬虫脚本和异常客户端}，不是用户。
  \item burst 给到 2$\times$ rps 的依据：
  \begin{itemize}
    \item 浏览器一个页面 30+ 静态 GET 是常态，瞬时 burst $\approx$ 100；
    \item burst 留 200 = 让一次正常加载\textbf{绝不触发}。
  \end{itemize}
  \item 这层是\textbf{兜底}：业务层的精细限流挂掉时，至少不会被压死。
\end{itemize}
\srcnote{routes/router.go:22 TokenBucket(rate.Limit(100), 200)}
\end{frame}

\begin{frame}{TokenBucket 算法：桶 + 令牌生成 + 漏出}
\begin{tikzpicture}[node distance=8mm and 14mm]
  \node[node-box,node-warn] (gen) {令牌生成\\100/s 稳定};
  \node[node-box,right=of gen] (bucket) {桶\\容量 200};
  \node[node-box,right=of bucket,node-ok] (out) {请求漏出};
  \node[node-box,below=of bucket,node-bad] (drop) {满了丢弃\\status=70001};

  \draw[arrow] (bucket) -- (out);

\begin{itemize}
  \item 令牌按 \textbf{rps} 匀速生成；桶满则溢出。
  \item 请求拿到令牌才放行，拿不到立刻返回 70001。
  \item burst 容量决定能否吸收\textbf{瞬时尖刺}（比如页面首屏并发）。
\end{itemize}

\end{frame}

\begin{frame}[fragile]{TokenBucket 中间件实现 (逐行)}
\begin{lstlisting}[language=Go,firstnumber=22,basicstyle=\ttfamily\tiny,
  caption={\srcnote{middleware/ratelimit.go:22--49}}]
func TokenBucket(rps rate.Limit, burst int) gin.HandlerFunc {
    var mu sync.Mutex
    limiters := make(map[string]*rate.Limiter)
    get := func(ip string) *rate.Limiter {
        mu.Lock(); defer mu.Unlock()
        if l, ok := limiters[ip]; ok { return l }
        l := rate.NewLimiter(rps, burst); limiters[ip] = l; return l
    }
    return func(c *gin.Context) {
        if !get(c.ClientIP()).Allow() {
            c.JSON(200, gin.H{"status": e.ErrRateLimitExceeded})
            c.Abort(); return
        }
        c.Next()
    }
}
\end{lstlisting}
\scriptsize ① map 按 IP 隔离；② 每 IP 一个 \texttt{rate.Limiter}；③ \texttt{Allow()} 非阻塞；④ 拒了直接 70001；⑤ 进程内 map 是单机代价，多实例要 Redis token bucket。
\end{frame}

\section{第二层：SlidingWindow 用户精确限流}
% ============================================================

\begin{frame}{秒杀为什么必须用 SlidingWindow}
\begin{itemize}
  \item 秒杀的\textbf{业务对手}是脚本，不是用户：
  \begin{itemize}
    \item 真人最快点击 \textbf{300--500ms 一次} $\to$ 1s 最多 2--3 次；
    \item 脚本可以 1s 打 1000 次。
  \end{itemize}
  \item 阈值定 \textbf{3/s/用户}：
  \begin{itemize}
    \item 真用户：留 50\% 余量给"双击 + 网络重试"；
    \item 黑产：第 4 次开始 70001 截断。
  \end{itemize}
  \item TokenBucket 不够：
  \begin{itemize}
    \item 必须按\textbf{用户 ID}（不是 IP）限，否则一个 NAT 出口下所有用户共享一个桶；
    \item 必须\textbf{分布式精确}，多实例下用 Redis 共享状态。
  \end{itemize}
\end{itemize}
\srcnote{routes/router.go:136--142 SlidingWindow Scope:seckill Limit:3 Window:1s ByUser:true}
\end{frame}

\begin{frame}[fragile]{SlidingWindow Lua 脚本 (逐行)}
\begin{lstlisting}[language=LuaRedis,firstnumber=18,basicstyle=\ttfamily\tiny,
  caption={\srcnote{repository/cache/ratelimit.go:18--34}}]
local key, window_ms = KEYS[1], tonumber(ARGV[1])
local limit, now_ms, member = tonumber(ARGV[2]), tonumber(ARGV[3]), ARGV[4]
redis.call('ZREMRANGEBYSCORE', key, 0, now_ms - window_ms)
local count = redis.call('ZCARD', key)
if count >= limit then return {0, count} end
redis.call('ZADD', key, now_ms, member)
redis.call('PEXPIRE', key, window_ms)
return {1, count + 1}
\end{lstlisting}
\small 逐行：① 入参 key/窗口/阈值/now/请求 id；② \textbf{ZREMRANGEBYSCORE} 滑掉过期（关键）；③ ZCARD 数当前窗口；④ 超阈值返回 0；⑤ 通过则 ZADD 入；⑥ PEXPIRE 自动回收；⑦ \textbf{整脚本单原子}，无竞态。
\end{frame}

\section{第三层：CircuitBreaker 保护慢依赖}
% ============================================================

\begin{frame}{熔断保护的不是自己，是下游}
\begin{itemize}
  \item gomall \texttt{/paydown} 的下游 = 第三方支付（Stripe / 微信 / 支付宝）。
  \item 这种依赖的\textbf{失败模式}是慢，不是 500：
  \begin{itemize}
    \item 正常 RT 200ms；
    \item 异常时 RT 30s 才超时；
    \item 不熔断 $\to$ goroutine 堆积 $\to$ 内存爆 $\to$ 自己跟着挂。
  \end{itemize}
  \item CircuitBreaker 的业务含义：\textbf{"我代下游回答你不行，免得你等 30 秒"}。
  \item 三态阈值 \textbf{5 失败 / 10s 等待 / 3 探测}（router.go:114--119）：
  \begin{itemize}
    \item 5 = 容忍偶发抖动，不会一抖就跳；
    \item 10s = 给下游一个完整告警 + 自愈周期；
    \item 3 = 探测窗口够小，避免 HalfOpen 也被打爆。
  \end{itemize}
\end{itemize}
\srcnote{routes/router.go:114--119 三态阈值}
\end{frame}

\begin{frame}[fragile]{CircuitBreaker.allow 状态转移 (逐行)}
\begin{lstlisting}[language=Go,firstnumber=76,basicstyle=\ttfamily\tiny,
  caption={\srcnote{middleware/circuitbreaker.go:76--106 allow}}]
func (cb *circuitBreaker) allow() error {
    switch circuitState(cb.state.Load()) {
    case stateClosed: return nil                          // 1) 关闭态零开销
    case stateOpen:
        if time.Now().UnixNano()-cb.openedAt.Load() < int64(cb.opt.OpenTimeout) {
            return errCircuitOpen                         // 2) Open 期间一律拒
        }
        cb.mu.Lock(); defer cb.mu.Unlock()
        if circuitState(cb.state.Load()) == stateOpen {   // 3) 锁内 double-check
            cb.state.Store(int32(stateHalfOpen))
            cb.halfOpenReq.Store(0)
        }
        fallthrough                                       // 4) 进入 HalfOpen
    case stateHalfOpen:
        if cb.halfOpenReq.Add(1) > cb.opt.HalfOpenMaxReq {
            return errCircuitOpen                         // 5) 探测超限退回
        }
        return nil
    }
    return nil
}
\end{lstlisting}
\scriptsize ① Closed 仅 atomic load；② Open 内不查时间就拒；③ Open$\to$HalfOpen 锁内 double-check；④ HalfOpen 用原子 Add 限并发；⑤ 热路径无 mutex。
\end{frame}

\begin{frame}[fragile]{report：决定下一刻是 Closed 还是 Open}
\begin{lstlisting}[language=Go,firstnumber=108,basicstyle=\ttfamily\tiny,
  caption={\srcnote{middleware/circuitbreaker.go:108--128 report}}]
func (cb *circuitBreaker) report(failed bool) {
    switch circuitState(cb.state.Load()) {
    case stateClosed:
        if failed {
            if cb.failures.Add(1) >= cb.opt.FailureThreshold {
                cb.tripOpen()             // 连续失败到阈值
            }
        } else { cb.failures.Store(0) }   // 成功重置
    case stateHalfOpen:
        if failed { cb.tripOpen()         // 任一失败 -> Open
        } else if cb.halfOpenReq.Load() >= cb.opt.HalfOpenMaxReq {
            cb.state.Store(int32(stateClosed)) // 全成功 -> Closed
            cb.failures.Store(0)
        }
    }
}
\end{lstlisting}
\scriptsize ① Closed 失败计数；② 成功重置避免偶发抖动跳闸；③ HalfOpen \textbf{非对称}——任一失败立即回 Open，必须全成功才回 Closed；④ "信任成本"对资金链路合适：宁保守不再打死下游。
\end{frame}

\section{第四层：异步下单削峰}
% ============================================================

\begin{frame}{双 11 0 点：100k QPS 怎么扛}
\begin{itemize}
  \item 同步下单链路：HTTP $\to$ ReserveStock $\to$ INSERT order $\to$ COMMIT $\to$ 回客户端，p99 大约 50--100ms。
  \item 100k QPS $\times$ 100ms = \textbf{1 万并发}进 MySQL，连接池打爆。
  \item gomall 的方案：把"DB 落库"从同步链路里拆出来。
  \begin{enumerate}
    \item 同步部分：reserve 库存 + 写 ticket + 投 MQ $\to$ 返回 ticket；
    \item 异步部分：consumer 慢慢消费 MQ $\to$ INSERT order $\to$ 回写 ticket 状态；
    \item 客户端：拿 ticket 轮询 \texttt{/orders/status}。
  \end{enumerate}
  \item 收益：DB 压力从"瞬时尖刺"变成"恒定 1k QPS 平流"。
\end{itemize}
\srcnote{service/order\_async.go:119--172 OrderEnqueue}
\end{frame}

\section{第五层：HTTP cache 把读路径 CDN 化}
% ============================================================

\begin{frame}{读流量的最优解：根本不到 gomall}

\begin{frame}{级联防护的两个隐藏陷阱}
\begin{itemize}
  \item \textbf{陷阱 1：阈值同步雪崩}。如果 CB-A 与 CB-B 都设 5/10s，下游恢复瞬间所有上游同时 HalfOpen $\to$ 探测流量乘 N 倍打回去 $\to$ 下游再次挂。\\
  对策：外层 CB 的 OpenTimeout \textbf{要长于}内层（如 30s vs 10s），错开探测窗口。
  \item \textbf{陷阱 2：兜底链路也会挂}。CB-C Open 时若兜底返回的 cache stale 也过期，CB-B 会把 cache 读失败计作业务失败 $\to$ 跟着 Open。\\
  对策：兜底路径\textbf{不能}计入 CB-B 的失败计数；report 时按 status code 维度分桶。
  \item gomall 当前只挂了 CB-A（paydown），CB-B / CB-C 是把单实例 CB 复制到下游服务上的扩展模式。
\end{itemize}
\srcnote{middleware/circuitbreaker.go:69 失败判定 = 5xx 或 c.Errors; 兜底路径需要在 handler 内吞掉 5xx 改成 200+stale}

\begin{frame}{大促容量评估：怎么算出"需要多少机器"}
\begin{itemize}
  \item 输入：① 业务侧预估的峰值 QPS；② 单机基线（gomall 64k RPS）；③ 安全系数（一般 1.5--2$\times$）。
  \item 计算：副本数 = $\lceil$ 预估峰值 $\times$ 安全系数 / 单机基线 $\rceil$。
  \item 双 11 例：100k QPS / 64k RPS $\times$ 1.5 = 2.34 $\to$ \textbf{至少 3 台}。
  \item 但秒杀 / 红包链路 \textbf{不是} 64k RPS 的水平：实测 52k 是 \texttt{/skill\_product/skill}，下单写链路 50k；要按\textbf{最弱链路}（这里是异步落库的 consumer 消费速度）算容量。
  \item 容量评估的常见错误：① 只看 RPS 不看 p99；② 没考虑下游放大（一个 paydown 触发 3 个下游调用）；③ 忽略冷启动（K8s 拉镜像 30s 起步）。
\end{itemize}

\begin{frame}{HPA 自动扩缩容：怎么定 metric 与冷却时间}
\begin{itemize}
  \item CPU-based HPA 是最朴素的：CPU > 70\% 扩、< 30\% 缩。问题：扩容前 30s 已经把请求拒了。
  \item 自定义 metric HPA：用 QPS / 队列堆积量 / 连接数作为触发条件，比 CPU 提前 1--2min 感知。
  \item gomall 异步下单的天然信号：MQ 队列堆积 > 10k $\to$ 扩 consumer。
  \item 关键：\textbf{冷却时间}（cooldown）—— 缩容窗口必须长于扩容（10min vs 1min），避免抖动来回拉。
  \item \textbf{预热与缓冲池}：新副本起来后先放 5\% 流量预热（JIT、连接池、缓存预填），10s 后切到 100\%，避免冷启动 p99 暴涨。
\end{itemize}

\begin{frame}{面试 Q\&A (1/4)}
\textbf{Q1}：100 RPS / 200 burst 这两个数字怎么定？\\
\hspace{4mm}\small A：基线 64k RPS / 单 IP 用不到 1/600，100 留给爬虫脚本做兜底；burst 200 容纳一次首屏并发 100+，不会误伤真用户。

\vspace{2mm}
\textbf{Q2}：为什么秒杀用 SlidingWindow 不用 TokenBucket？\\
\hspace{4mm}\small A：① 必须按用户而不是 IP，否则 NAT 出口全员共享；② 必须分布式精确，多实例下用 Redis 共享状态；③ ZSet 比 fixed-window 没有窗口边界可以利用。

\vspace{2mm}
\textbf{Q3}：CircuitBreaker 5/10s/3 三个数字有什么讲究？\\
\hspace{4mm}\small A：5 = 容忍偶发抖动不一抖就跳；10s = 给下游一次完整告警+自愈周期；3 = HalfOpen 探测够小，避免再次打死下游。
\end{frame}

\begin{frame}{面试 Q\&A (2/4)}
\textbf{Q4}：异步下单是不是会丢单？\\
\hspace{4mm}\small A：不会。库存先 Reserve（Redis Lua 原子），任一失败都 ReleaseReservation 回滚；ticket TTL=1h 兜底；MQ 用 RabbitMQ 持久化 + 手动 ack。代码位置 \texttt{service/order\_async.go:119--172}。

\vspace{2mm}
\textbf{Q5}：为什么熔断返回 200 不返回 503？\\
\hspace{4mm}\small A：app 端老网关不能优雅处理 5xx；统一走业务码（70002）= 客户端只用维护一套 status switch；监控不要按 4xx/5xx 算错误率，必须看 status。代码位置 \texttt{middleware/circuitbreaker.go:57--65}。

\vspace{2mm}
\textbf{Q6}：HTTP cache 的 max-age 怎么定？\\
\hspace{4mm}\small A：取决于"改价 / 下架"的最大可接受延迟。商品详情 60s，分类 / 轮播 300s，列表 30s（库存敏感）。代码位置 \texttt{routes/router.go:39--45}。
\end{frame}

\begin{frame}{面试 Q\&A (3/4)}
\textbf{Q7}：多层熔断会不会"探测雪崩"？怎么办？\\
\hspace{4mm}\small A：会。外层 CB 的 OpenTimeout 要长于内层（30s vs 10s）错开 HalfOpen 探测窗口；同时兜底路径不应计入失败计数，否则 cache 过期会让外层跟着 Open。代码位置 \texttt{middleware/circuitbreaker.go:69 失败判定}。

\vspace{2mm}
\textbf{Q8}：全链路压测怎么避免污染真实数据？\\
\hspace{4mm}\small A：流量染色：网关层注入 \texttt{X-Test} header $\to$ 透传到 ctx $\to$ DAO 看 ctx 切 \texttt{order\_shadow} 表；MQ 用独立 routing-key 走影子队列；Redis 用 \texttt{shadow:} 前缀隔离。代码位置 \texttt{middleware/coloring.go（扩展）}。

\vspace{2mm}
\textbf{Q9}：灰度发布按什么维度切流最合理？\\
\hspace{4mm}\small A：① 内部 IP 段——蓝绿冒烟；② userID hash——保证"同一用户看同一版"，避免购物车看到一半的数据；③ 按比例随机——只适合无状态读路径。写路径必须按 userID 锁定。
\end{frame}

\begin{frame}{面试 Q\&A (4/4)}
\textbf{Q10}：限流算法为什么不选自适应（Sentinel BBR）？\\
\hspace{4mm}\small A：自适应是黑盒、冷启动 5--10min 才稳；秒杀 3/秒/用户 这种阈值用业务语言能算清楚，没必要让运维去看 BBR 曲线判断。代码位置 \texttt{repository/cache/ratelimit.go:18--34}。

\vspace{2mm}
\textbf{Q11}：HPA 用什么 metric 比 CPU 更合适？\\
\hspace{4mm}\small A：异步下单链路用 \textbf{MQ 队列堆积}：堆积 > 10k 扩 consumer，比等 CPU 飙高快 1--2min；同步链路看 inflight 请求数 / p99 延迟；cooldown 缩容窗口要长于扩容（10min vs 1min）避免抖动。
\end{frame}
