\section{为什么单桶 stock 不够用}
% ============================================================

\begin{frame}{下单到付款，中间那 15 分钟}
\begin{itemize}
  \item 用户点"提交订单"到点"立即支付"，gomall 给 15 分钟。
  \item 这 15 分钟里：商品\textbf{不能再被别人下单}，但也\textbf{还没真正卖出}。
  \item 单桶 \texttt{stock=N} 模型无法表达这个"占着但没消耗"的中间态：
  \begin{itemize}
    \item 一减就减，用户不付钱也回不来（库存丢）；
    \item 不减，并发下两个人同时下同一件（超卖）。
  \end{itemize}
\end{itemize}
\srcnote{repository/cache/inventory.go:11--14 注释明确两桶定位}
\end{frame}

\begin{frame}{业务问题清单（本 deck 七连问）}
\begin{enumerate}
  \item 为什么把库存切成 \textbf{available + reserved} 两桶？
  \item 抢库存为什么必须走 Redis Lua，不走 DB \texttt{SELECT FOR UPDATE}？
  \item Redis 重启之后，库存数字从哪儿来？
  \item 下单成功，但 DB 事务回滚，预占的库存怎么办？
  \item 关单（超时 / 用户主动）触发的库存退回，落在哪一行代码？
  \item 客服收到"明明显示有货，下单却失败"的工单，怎么回答？
  \item 业务到底承诺什么 —— 超卖能赔多少钱、误锁赔多少钱？
\end{enumerate}
\srcnote{对应代码 repository/cache/inventory.go + service/inventory/syncer.go + service/order.go + service/order\_cancel.go}
\end{frame}

\vspace{1mm}
\begin{itemize}\setlength\itemsep{-1mm}
  \item \textbf{reserve}：available -= n, reserved += n（下单时调）
  \item \textbf{commit}：reserved -= n（支付成功，库存真正消耗）
  \item \textbf{release}：reserved -= n, available += n（取消 / 关单）
\end{itemize}
\small in\_transit / defective 归 deck 09 + WMS（仓储管理系统，本仓库未实现）。
\srcnote{repository/cache/inventory.go:29--66 三段 Lua 脚本完整定义}
\end{frame}

\begin{frame}[fragile]{reserveScript：8 行 Lua，零超卖的全部秘密}
\begin{lstlisting}[language=LuaRedis,firstnumber=32,
  caption={\srcnote{repository/cache/inventory.go:32--44 reserveScript}}]
local avail = redis.call('GET', KEYS[1])    -- 1. 读 available
if avail == false then
    return -2                                -- 2. key 不存在 = 库存未初始化
end
local need = tonumber(ARGV[1])
if tonumber(avail) < need then
    return -1                                -- 3. 不足 = ErrStockInsufficient
end
redis.call('DECRBY', KEYS[1], need)          -- 4. available -= n
redis.call('INCRBY', KEYS[2], need)          -- 5. reserved  += n
return 1                                     -- 6. 成功
\end{lstlisting}
\small \textbf{为什么必须打成一个 Lua}：脚本对 Redis 是单命令，redis-server 单线程从头执行到尾，中途\textbf{不会插入}其它客户端的 GET / DECRBY / INCRBY。如果分成 3 次 RTT 做 check-then-act，500 并发立刻超卖。
\end{frame}

\begin{frame}[fragile]{commitScript / releaseScript：成对出现，对称设计}
\begin{lstlisting}[language=LuaRedis,firstnumber=48,
  caption={\srcnote{repository/cache/inventory.go:48--66 commit + release}}]
-- commitScript：支付成功，reserved 真正消耗
local r = redis.call('GET', KEYS[1])           -- KEYS[1]=reserved
if r == false or tonumber(r) < tonumber(ARGV[1]) then
    return -1
end
redis.call('DECRBY', KEYS[1], ARGV[1])         -- reserved -= n
return 1
-- releaseScript：取消，reserved 退回 available
local r = redis.call('GET', KEYS[1])           -- KEYS[1]=reserved
if r == false or tonumber(r) < tonumber(ARGV[1]) then
    return -1
end
redis.call('DECRBY', KEYS[1], ARGV[1])         -- reserved -= n
redis.call('INCRBY', KEYS[2], ARGV[1])         -- available += n
return 1
\end{lstlisting}
\small commit 只动 reserved（钱已付，不退）；release 两桶都动。check $r \geq n$ 避免 reserved 跌负。
\end{frame}

\begin{frame}{Redis Lua vs DB \texttt{SELECT FOR UPDATE}：实测尾延迟}
\begin{tabular}{@{}lrr@{}}
\toprule
& Redis Lua & DB 悲观锁 \\
\midrule
avg 延迟    & 1.24ms  & 1.29ms \\
p95 延迟    & 3.52ms  & 3.65ms \\
\textbf{max 延迟} & \textbf{136ms} & \textbf{453ms} \\
RPS         & 51{,}362 & 50{,}142 \\
\bottomrule
\end{tabular}
\vspace{2mm}
\small avg / p95 看不出差距，但 \textbf{max 拉开 3.3 倍}。原因：
\begin{itemize}
  \item DB：BEGIN $\to$ \texttt{SELECT \dots FOR UPDATE} (行锁) $\to$ 业务 check $\to$ UPDATE $\to$ COMMIT，至少 4 次 RTT + 行锁串行；
  \item Redis Lua：\textbf{1 次 RTT}，单线程跑完 5 条命令。
\end{itemize}
\srcnote{stressTest/REPORT.md \S 6 优惠券：Redis Lua vs DB 锁}
\end{frame}

\begin{frame}{三方案对比：Redis Lua vs DB FOR UPDATE vs 分布式锁}
\begin{tabular}{@{}lp{3.2cm}p{3.0cm}p{3.2cm}@{}}
\toprule
& Redis Lua 单实例 & DB 行锁 & ZK / etcd 分布式锁 \\
\midrule
原子性来源    & redis-server 单线程       & InnoDB 行锁         & ZAB / Raft 共识 \\
单次开销      & 1 RTT $\sim$ 1ms          & 4 RTT + 锁等待       & 1--3 次共识广播 \\
水平扩展      & 分 slot / 分实例（业务分片） & 按主键分表 / 分库      & 节点数加倍后写性能不增 \\
故障恢复      & AOF + SeedFromDB         & 主从切换 + Redo log   & 选主 < 10s \\
\textbf{尾延迟}  & 136ms（实测）            & \textbf{453ms}（实测）& 秒级（选主期间） \\
\textbf{超卖风险} & 单实例 0（Lua 原子）       & 0（事务 + 锁）       & 0（共识 quorum） \\
\bottomrule
\end{tabular}
\vspace{2mm}
\small gomall 选 \textbf{Redis Lua 单实例}：日单峰值 10w 级，单 Redis 足够；分布式锁的"选主期间不可写"对秒杀不可接受。Lua 的"单点"风险用 AOF + SeedFromDB 兜底（见 \S 启动 SeedFromDB）。
\srcnote{stressTest/REPORT.md \S 6 + repository/cache/inventory.go:29--66}
\end{frame}

\begin{frame}[fragile]{反例：DB \texttt{SELECT FOR UPDATE} 写法（选型对照）}
\begin{lstlisting}[language=Go,firstnumber=1,
  caption={\srcnote{对照写法 —— 仓库未采用，仅做选型对照}}]
// 反面：把扣库存放进 DB 事务，悲观锁串行化
err := dao.NewDBClient(ctx).Transaction(func(tx *gorm.DB) error {
    var p model.Product
    // 行锁 + 业务 check + UPDATE，至少 3 次 RTT
    if e := tx.Set("gorm:query_option", "FOR UPDATE").
        First(&p, productID).Error; e != nil { return e }
    if p.Num < int(num) {
        return ErrStockInsufficient
    }
    return tx.Model(&p).Update("num", p.Num-int(num)).Error
})
\end{lstlisting}
\small \textbf{三个致命问题}：(1) 表达不了 reserved 中间态；(2) 行锁串行，TP99 秒级（实测 453ms max）；(3) DB 是真理之源，热写打进 DB = 把副作用集中到最贵的资源。
\end{frame}

\begin{frame}{套餐 SKU：1 单同时扣 N 个底层 SKU}

  \begin{itemize}

\begin{frame}{智能选仓三档 + 跨仓原子性}
\textbf{选仓策略}：
\begin{enumerate}
  \item \textbf{就近优先}：用户地址 $\to$ 省/市 $\to$ 默认仓表；不足降级第二近。
  \item \textbf{凑单优先}：多 SKU 单优先选能凑齐所有 SKU 的仓，避免拆单（拆单 = 双倍物流费 + 体验差）。
  \item \textbf{兜底全国仓}：就近 / 凑单都不足时走中央仓（成本高 + 时效慢，但保下单成功）。
\end{enumerate}
\vspace{1mm}
\textbf{跨仓扣减的原子性}：单仓 Lua 无问题；跨仓凑单（华北 1 件 + 华东 1 件）需把所有仓 SKU key 用 hashtag \texttt{\{sku\_id\}} 强制路由到\textbf{同一 Redis Cluster slot}，Lua 即可跨 key 原子操作。代价：单 slot 写入吞吐有上限，运维复杂度 = 1（远胜 2PC）。
\srcnote{Redis Cluster hashtag 规则；同 slot 才能跨 key Lua}
\end{frame}

\begin{frame}[fragile]{SeedFromDB 主循环：分页 + EXISTS 双保险}
\begin{lstlisting}[language=Go,firstnumber=14,
  caption={\srcnote{service/inventory/syncer.go:14--48 SeedFromDB}}]
func SeedFromDB(ctx context.Context, batchSize int) error {
    if batchSize <= 0 { batchSize = 500 }            // 默认页 500
    d := dao.NewDBClient(ctx); var lastID uint
    for {
        var rows []*model.Product
        q := d.Model(&model.Product{}).Order("id ASC").Limit(batchSize)
        if lastID > 0 { q = q.Where("id > ?", lastID) }  // 游标分页
        if err := q.Find(&rows).Error; err != nil { return err }
        if len(rows) == 0 { break }
        for _, p := range rows {
            lastID = p.ID
            exists, _ := cache.RedisClient.Exists(ctx,
                cache.StockAvailableKey(p.ID)).Result()
            if exists == 1 { continue }              // 跳过运行期数据
            cache.InitStock(ctx, p.ID, int64(p.Num)) // 覆盖写 0
        }
    }
    return nil
}
\end{lstlisting}
\end{frame}

\begin{frame}{为什么是"游标分页 + EXISTS 跳过"}
\begin{itemize}
  \item \textbf{游标分页}（\texttt{WHERE id > lastID LIMIT 500}）：1M 行的 product 表 \texttt{OFFSET 999500} 要扫近百万行；游标走 PK 索引，每页常数时间，启动期 DB CPU 不抖。
  \item \textbf{EXISTS 检查 / 跳过}（line 32--40 是关键）：
  \begin{itemize}
    \item 冷启动：available key 不存在 $\to$ 写入 DB 水位；
    \item 滚动重启 / 多副本：available 已被前一实例写过 $\to$ 跳过，\textbf{不抹平运行期 reserve}；
    \item \texttt{InitStock} 是\textbf{覆盖写}（\texttt{SET k v 0}），没 EXISTS 就是事故。
  \end{itemize}
\end{itemize}
\srcnote{service/inventory/syncer.go:34--41 EXISTS 跳过}
\end{frame}

\begin{frame}{库存预热：大促前的全量灌入}
\begin{itemize}
  \item \textbf{冷启动}（机房新部署）：SeedFromDB 跑一次，1M SKU $\times$ 500/批 = 2000 批 $\times$ 50ms $\approx$ 100s。
  \item \textbf{大促预热}（双 11 提前一天）：手动触发 \texttt{SeedFromDB(ctx, 1000)} 灌全量；抽样 1000 SKU 对比 DB num 与 Redis avail；活动前 5min 关商家改库存入口，避免双写竞态。
  \item \textbf{分批 + 灰度}：1000 SKU/批 + 每批 10ms sleep；先灌 1\% $\to$ 10\% $\to$ 100\%，每档观察 5min。
\end{itemize}
\srcnote{service/inventory/syncer.go:14 SeedFromDB；预热可拆独立工具 cmd/inventory\_seed/main.go}

\begin{frame}{面试 Q\&A (1/3)：基础设计 + 容灾}
\textbf{Q1}：为什么不能用单条 \texttt{UPDATE product SET num=num-1 WHERE num>0}？\\
\hspace{4mm}\small A：单 SQL 防得了超卖，但表达不了"占着未付"。下单到支付 15min，单桶只能选"立刻扣"（不付就丢）或"不扣"（并发超卖），都不可接受。代码：\texttt{inventory.go:11--14}。

\vspace{1mm}
\textbf{Q2}：Redis 挂了 30 秒，恢复后库存怎么办？\\
\hspace{4mm}\small A：AOF / RDB 保住大头；丢失部分由 SeedFromDB 重启时从 \texttt{product.num} 回灌；运行期 reserved 通过 EXISTS 跳过保护；差额走巡检脚本。代码：\texttt{syncer.go:14--48}。

\vspace{1mm}
\textbf{Q3}：DB 事务成功但 Redis commit 失败怎么办？\\
\hspace{4mm}\small A：outbox 事件落库后消费者重试 CommitReservation；reserved 偏高由巡检 leak 报警接管。属"业务可接受 + 监控可发现"的取舍。代码：\texttt{service/order.go:64--78}。

\vspace{1mm}
\textbf{Q4}：100 万 SKU 启动要多久？内存多少？\\
\hspace{4mm}\small A：每 SKU 两桶 $\times$ 80B = 160 MB（16GB 实例 1\%）；冷启动 500/批 $\times$ 50ms = 100s。期间下单返 ErrStockNotInit 触发降级 banner。
\end{frame}

\begin{frame}{面试 Q\&A (2/3)：SKU / 多仓 / 选型 / 秒杀}
\textbf{Q5}：多规格（颜色、尺寸）怎么扩展？\\
\hspace{4mm}\small A：Redis key 从 \texttt{stock:available:\{pid\}} 升级到 \texttt{\{sku\_id\}}，Lua 一字不改；难点在订单表迁移（老订单只有 product\_id），需双写过渡期 + 回填脚本。

\vspace{1mm}
\textbf{Q6}：跨仓凑单怎么保原子性？\\
\hspace{4mm}\small A：所有仓 SKU 的 key 用 Cluster hashtag \texttt{\{sku\_id\}} 强制路由到同一 slot，Lua 同 slot 内原子操作多 key。代价：单 slot 写入吞吐受限，运维复杂度 = 1（远胜 2PC）。

\vspace{1mm}
\textbf{Q7}：为什么不上 ZK / etcd 分布式锁？\\
\hspace{4mm}\small A：分布式锁基于共识（ZAB / Raft），单次 1--3 次广播比 Lua 慢一个数量级；选主期间（5--10s）不可写，秒杀不可接受。日单峰 10w 级单 Redis 足够。论证：\texttt{stressTest/REPORT.md \S 6}。

\vspace{1mm}
\textbf{Q8}：秒杀为什么要分 Redis 实例？\\
\hspace{4mm}\small A：Redis 单线程，一个慢 Lua stall 所有客户端。秒杀 10w QPS 会把主集群 CPU 占满，影响商详 / 购物车 / 下单。秒杀实例关 AOF（结束销毁）+ TCP backlog 32768，吞吐翻倍。
\end{frame}

\begin{frame}{面试 Q\&A (3/3)：防黄牛 / 应急 / 巡检 / 并发证据}
\textbf{Q9}：怎么防黄牛批量下单？\\
\hspace{4mm}\small A：四层防线 —— 单用户上限（Lua 加 \texttt{owned $<$ perUser} check）+ 设备指纹 + 邀请码限购 + 实名认证。每层把黄牛成本拉高 10\%，劝退而非封死。模板：\texttt{coupon.go:30--44 ClaimCouponAtomic}。

\vspace{1mm}
\textbf{Q10}：发生超卖了，怎么应急？\\
\hspace{4mm}\small A：4 小时应急 SOP —— T+5min 锁单（关商详页下单入口）、T+15min 跑对账脚本定位范围、T+30min 发客服话术、T+1h 发补偿券。技术侧关键：\textbf{下单入口要动态可关}，不能等发版。

\vspace{1mm}
\textbf{Q11}：怎么做库存巡检？\\
\hspace{4mm}\small A：恒等式 \texttt{available + reserved + committed = product.num}，前两项来自 \texttt{GetStockSnapshot}，committed = \texttt{SUM(num) WHERE type=Paid}。偏差 1\% P3、5\% P1。代码：\texttt{inventory.go:127--138}。

\vspace{1mm}
\textbf{Q12}：500 并发抢 100 件，怎么证明真不超卖？\\
\hspace{4mm}\small A：\texttt{TestInventory\_NoOversellUnderConcurrency} 写死 \texttt{success == exactly 100}，少卖 / 超卖 CI 都红。本质是 Redis 单线程 + Lua 原子。代码：\texttt{inventory\_test.go:52--82}。
